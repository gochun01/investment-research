# psf-monitor 자율 판단 보고서 — 작동 원리

> state.json → 4-Phase 파이프라인 → 적응형 HTML 보고서.
> 고정 템플릿이 아니다. 데이터가 보고서의 형태를 결정한다.
> reaction-monitor의 render_adaptive.py에서 진화한 PSF 전용 모듈.

---

## 1. 핵심 철학

```
고정 구조의 문제:
├── 데이터 적은데 11개 섹션 → 빈 섹션, 억지 채움
├── 데이터가 한쪽에 집중인데 균등 배분 → 핵심 희석
├── 매일 다른 데이터인데 같은 틀 → 맞지 않는 옷
└── 결과: 형식은 완벽하나 내용이 형식을 따라감

자율 구조의 원칙:
├── 데이터를 먼저 읽는다
├── 데이터의 무게 중심을 찾는다
├── 무게 중심에 맞게 구조를 설계한다
├── 데이터가 부족한 영역은 과장하지 않고 인정한다
└── 결과: 내용이 형식을 결정함
```


## 3. 트리거 정의

```
"보여줘" = 현재형
  state.json (이미 있는 데이터) → 바로 Phase 1~4 → HTML
  새 관측 없음. 있는 그대로 렌더링.
  → python core/render_adaptive.py

"만들어줘" = 갱신형
  SKILL-core.md D Loop 실행 (MCP 수집 → axis-map → 판정)
  → state.json 갱신
  → Phase 1~4 → HTML
  → python core/render_adaptive.py

"설계만 보여줘"
  Phase 1~2만 실행. HTML 미생성. 설계안(유형+섹션) 출력.
  → python core/render_adaptive.py --design-only

"투영 보고서 보여줘"
  projection.json → Phase 1~4 → HTML
  → python core/render_adaptive.py --projection
```

## 4. 4-Phase 파이프라인

### Phase 1: Extract Five — 5가지 추출

```
state.json에서 5가지 핵심 요소만 추출한다.
구조를 외우려 하지 말고 이 5가지만 뽑는다.

┌────────────────┬──────────────────────────────────────────────────────┐
│ 요소            │ state.json 매핑                                      │
├────────────────┼──────────────────────────────────────────────────────┤
│ ① Core Claim   │ regime + observations[0].signal                      │
│ (핵심 주장)     │ "🟡 경계 (다소 완화) — 5일 군사 정지로 L3 감속"      │
│                │ → 보고서의 첫 줄이 된다.                              │
├────────────────┼──────────────────────────────────────────────────────┤
│ ② Tension      │ divergences[] (존재 여부 + 건수 + 상세)               │
│ (긴장)         │ 있으면 → 보고서 구조가 대립형(A) 또는 복합형(E)       │
│                │ 없으면 → 스냅샷형(C) 또는 분기형(D)                   │
├────────────────┼──────────────────────────────────────────────────────┤
│ ③ Gravity      │ observations[].severity 분포 집계                    │
│ (무게 중심)     │ critical: N건 → large 섹션                           │
│                │ high: N건 → medium 섹션                              │
│                │ medium: N건 → small 섹션                             │
│                │ low: N건 → 1줄 또는 생략                             │
│                │ → 무게 중심이 섹션 순서와 크기를 결정.                 │
├────────────────┼──────────────────────────────────────────────────────┤
│ ④ Timeline     │ next_questions에 status=="open"이 있는가?            │
│ (시간축)        │ 있으면 → "현재 상태 + 미래 분기"                     │
│                │ 없으면 → "현재 상태"                                  │
│                │ → 미래 분기가 있으면 시나리오 섹션 포함.               │
├────────────────┼──────────────────────────────────────────────────────┤
│ ⑤ Unresolved   │ next_questions(open) 건수 + unclassified[] 건수      │
│ (미해소)        │ > 0이면 → Open Questions 섹션 포함                   │
│                │ = 0이면 → 해당 섹션 생략                              │
└────────────────┴──────────────────────────────────────────────────────┘

부산물:
  active_links: links에서 "활성" 또는 "접근" 상태인 Link 목록
  → Link 상태 섹션 포함 여부 결정
```

### Phase 1 출력 예시 (내부용, 보고서에 미포함)

```
[Phase 1] Core Claim: 🟡 경계 (다소 완화) — 미국 5일 군사행동 정지
[Phase 1] Tension: 있음 (2건) — P↔S 불일치, F5 내부 괴리
[Phase 1] Gravity: critical 1 / high 2 / medium 2 / low 1
[Phase 1] Timeline: 현재 상태 + 미래 분기
[Phase 1] Unresolved: 6건
```

### Phase 2: Design Report — 설계

5가지 추출 결과로 보고서의 형태를 결정한다.

#### 2-A. 유형 판별 (A~E)

```
┌──────────────────────────────────────────────────────────┐
│                                                          │
│  Tension 있음?  ───YES───→  Timeline 미래?               │
│       │                      │YES → E 복합형             │
│       │                      │NO  → A 대립형             │
│       NO                                                 │
│       │                                                  │
│       └──→  Timeline 미래?                               │
│              │YES → D 분기형                              │
│              │NO  → C 스냅샷형                            │
│                                                          │
└──────────────────────────────────────────────────────────┘

유형별 특징:
  A 대립형: Clash 섹션 중심. 충돌하는 힘 시각화.
  C 스냅샷형: 대시보드 중심. 텍스트 최소, 수치 중심. 가장 가벼움.
  D 분기형: 시나리오 그리드 중심. 미래 경로 비교.
  E 복합형: 대립 + 분기 모두. 가장 무거움.
```

#### 2-B. 분류 판별 (report-class)

```
regime에 🔴 또는 L8 활성  → CRISIS ALERT (topbar 빨강)
시나리오 분기 또는 Tension → SPECIAL REPORT (topbar 파랑)
기본                       → RESEARCH NOTE (topbar 네이비)
--projection 플래그        → STRATEGY UPDATE (topbar 골드)
```

#### 2-C. 섹션 자율 구성

```
고정 섹션이 없다. Gravity(무게 중심)에 따라 구성한다.

항상 포함:
  [Large]  Executive Verdict (exec-box)     ← Core Claim
  [Large]  국면 대시보드 (regime-dashboard)  ← P/S/F 3열

조건부 포함:
  [Large]  긴장 구조       ← Tension 있을 때
  [Large]  핵심 관측       ← critical severity 있을 때
  [Medium] 주요 변화       ← high severity 있을 때
  [Medium] Link 상태       ← active link 있을 때
  [Medium] 축 상태         ← 항상 (axis-status)
  [Small]  보조 관측       ← medium severity 있을 때
  [Medium] 감시 + 시나리오 ← open question 있을 때
  [Small]  미해소 질문     ← unresolved > 0일 때

데이터 없는 영역 → 만들지 않는다. 억지 채움 금지.
섹션 수: 데이터가 결정. 4개일 수도, 10개일 수도 있다.
섹션 순서: 중요도순 (Large → Medium → Small).
```

#### Phase 2 출력 예시

```
[Phase 2] 유형: E 복합형 (대립 + 분기)
[Phase 2] 분류: SPECIAL REPORT
[Phase 2] 섹션: 10개
  [L] Executive Verdict
  [L] 국면 대시보드
  [L] 긴장 구조
  [L] 핵심 관측
  [M] 주요 변화
  [M] Link 상태
  [M] 축 상태
  [S] 보조 관측
  [M] 감시 + 시나리오
  [S] 미해소 질문
```

### Phase 3: Render HTML — 생성

설계안을 HTML로 조합한다.

#### 3-A. 디자인 시스템

```
CSS: assets/template-base.html
  ├── CSS 변수 (색상, 레짐 색상, 폰트)
  ├── 컴포넌트 클래스 (.topbar, .exec-box, .card, .link-grid, ...)
  ├── 반응형 (max-width: 780px)
  └── 다크 테마 (#0d1117 배경)

폰트:
  본문: Noto Sans KR
  수치: DM Mono

레짐 뱃지 색상:
  🟢 = #003d00 배경 + #4caf50 텍스트
  🟡 = #4a3800 배경 + #ffd600 텍스트
  🔴 = #3d0000 배경 + #ff5252 텍스트
  키워드 = #2a1a3e 배경 + #ce93d8 텍스트
```

#### 3-B. 컴포넌트 → state.json 매핑

```
┌──────────────────┬─────────────────────────────────────────────┐
│ 컴포넌트          │ state.json 데이터 원천                       │
├──────────────────┼─────────────────────────────────────────────┤
│ topbar           │ Phase 2의 report_class + class_color        │
│ report-header    │ last_updated + regime + macro_interface     │
│ exec-box         │ Five.core_claim + observations[0].path     │
│ regime-dashboard │ plates{} + structure{} + flow{}             │
│ divergence-card  │ divergences[]                               │
│ observation-card │ observations[] (severity별 필터)             │
│ link-status      │ links{} (active/approaching만)              │
```

#### 3-C. HTML 구조

```html
<!DOCTYPE html>
<head>
  <style> /* CSS 인라인. 외부 의존 = Google Fonts만 */ </style>
</head>
<body>
  <div class="topbar {color}">{REPORT_CLASS}</div>
  <div class="container">
    <div class="report-header">

      <div class="badges">{regime} {macro} {alignment}</div>
    </div>

    <!-- Phase 2에서 결정된 섹션들 순서대로 -->
    <div class="section-large">Executive Verdict</div>

    ...
    <div class="section-small">미해소 질문</div>

    <div class="footer">MCP {N}건 · 비율 {N%} · disclaimer</div>
  </div>

```

#### 3-D. 글쓰기 원칙

```
1. 첫 문장이 결론이다
   섹션마다 첫 문장 = 그 섹션의 결론. 뒤 문장은 근거.
   "배경을 먼저 설명하고 결론을 나중에" ✗

2. 수치가 먼저, 해석이 뒤에
   "BTC $69,933 (-1.2%) — SEC 발표 후 $75K 돌파 실패" ◯
   "최근 하락세를 보이고 있는 BTC는..." ✗

3. 확신과 추정을 구분한다
   데이터 기반 → 단정문. 추론 기반 → "~할 경우", "~가능성".

4. 빈 공간은 미덕이다
   데이터가 3줄이면 3줄로 끝낸다. 10줄로 늘리지 않는다.

5. 독자는 바쁘다
   스크롤 최소화. 핵심은 상단. 전체를 안 읽어도 판단 가능.
```

### Phase 4: Verify — 자기 검증

```
렌더링 완료 후 저장 전에 5가지를 검증한다.

V1. Core Claim 정합성
    "첫 화면의 Core Claim이 HTML에 존재하는가?"
    ✅ → 통과
    ❌ → FAIL. 재생성 1회 시도.

V2. 데이터 충실도
    "상위 3개 관측의 핵심 수치가 HTML에 반영되었는가?"
    ✅ → 통과
    ❌ → WARN. 로그 출력.

V3. 빈 섹션 검사
    "데이터 없이 만들어진 섹션이 있는가?"
    ✅ → 없음 (Phase 2에서 이미 제거)
    ❌ → WARN. 해당 섹션 삭제.

V4. 무게 비례
    "critical 관측이 large 섹션에 있는가? (Gravity 비례)"
    ✅ → 통과
    ❌ → WARN. 로그 출력.

V5. 독자 테스트
    "첫 화면(스크롤 없이)에 exec-box가 있는가?"
    ✅ → 통과
    ❌ → WARN. exec-box를 상단으로 이동.
```

## 6. 코드 구조 (core/render_adaptive.py)

```
render_adaptive.py (687줄)
│
├── extract_five(state) → dict
│   state.json → {core_claim, tension, gravity, timeline, unresolved, active_links}
│
├── design_report(five, state) → dict
│   Five → {type, type_name, report_class, class_color, sections, section_count}
│
├── render_html(design, state, five) → str
│   design + state → 완성 HTML 문자열
│   ├── HTML head + inline CSS
│   ├── topbar
│   ├── report-header + badges
│   ├── for section in design.sections:
│   │     render_section(section, state, five)
│   └── footer + disclaimer
│
├── render_section(sec, state, five) → str
│   섹션 ID별 분기:
│   ├── "exec"       → exec-box (Core Claim + path)
│   ├── "dashboard"  → render_dashboard() (P/S/F 3열)
│   ├── "clash"      → divergence-card (괴리)
│   ├── "critical"   → observation-card (critical)
│   ├── "high"       → observation-card (high)
│   ├── "links"      → link-grid (active/approaching)
│   ├── "axis"       → axis-grid (건재/감속/훼손)
│   ├── "medium"     → monitor-table (medium)
│   ├── "scenarios"  → table (next_questions)
│   └── "unresolved" → question-list (questions + unclassified)
│
├── render_dashboard(state) → str
│   plates{} + structure{} + flow{} → P/S/F 3컬럼 카드
│
├── verify(html, state, five) → list[str]
│   V1~V5 검증. 이슈 있으면 리스트 반환.
│
└── main()
    CLI 파싱 → load → Phase 1~4 → save
```

## 8. 도구 체인

```bash
# 현재 state → HTML (보여줘)
python core/render_adaptive.py

# 설계안만 확인
python core/render_adaptive.py --design-only

# 투영 보고서
python core/render_adaptive.py --projection

# 특정 히스토리 파일 → HTML
python core/render_adaptive.py --file history/2026-03-23.json

# 전체 파이프라인 (만들어줘)
# 1. PSF D Loop 실행 (state.json 갱신)
# 2. python core/validate.py
# 3. python core/render_adaptive.py
# 4. python core/snapshot.py save
# 5. python core/autonomy.py scan
```

## 9. F-테이블 (실패 패턴)

```
┌──────┬──────────────────────┬──────────────────────┬──────────────────────────┐
│ 코드  │ 실패 패턴             │ 원인                  │ 방어                      │
├──────┼──────────────────────┼──────────────────────┼──────────────────────────┤
│ F-01 │ state.json 없음       │ 파일 미존재/경로 오류  │ 오류 출력 + 종료.         │
│      │                      │                      │ "PSF 갱신 먼저" 권고.     │
├──────┼──────────────────────┼──────────────────────┼──────────────────────────┤
│ F-02 │ observations 비어있음 │ 수집 실패 / 빈 state  │ Core Claim = "관측 없음". │
│      │                      │                      │ 스냅샷형(C) 강제.         │
├──────┼──────────────────────┼──────────────────────┼──────────────────────────┤
│ F-03 │ HTML 렌더링 실패      │ 키 누락, 타입 오류     │ try-except 해당 섹션 스킵.│
│      │                      │                      │ 나머지 섹션 계속 렌더.    │
├──────┼──────────────────────┼──────────────────────┼──────────────────────────┤
│ F-04 │ V1 FAIL              │ Core Claim 미반영     │ 재생성 1회. 재실패 시     │
│      │ (Core Claim 없음)    │                      │ 로그 + 저장.             │
├──────┼──────────────────────┼──────────────────────┼──────────────────────────┤
│ F-05 │ 빈 섹션 존재          │ 조건부 섹션 로직 오류  │ V3에서 감지 → 삭제.       │
├──────┼──────────────────────┼──────────────────────┼──────────────────────────┤
│ F-06 │ Gravity 무시          │ 모든 섹션 동일 크기    │ V4에서 감지 → 경고.       │
│      │                      │                      │ critical → large 강제.    │
└──────┴──────────────────────┴──────────────────────┴──────────────────────────┘
```

## 10. 불변 규칙

```
1. 데이터가 구조를 결정한다 (구조가 데이터를 끼워 맞추지 않는다).
2. 데이터 없는 섹션을 만들지 않는다.
3. 첫 화면에 Core Claim이 있다.
4. 확신과 추정을 구분한다.
5. V1~V5 자기 검증을 생략하지 않는다.
6. PSF는 관측한다. 해석하지 않는다.
   "왜 그런지", "뭘 해야 하는지"를 보고서에 쓰지 않는다.
7. 출처를 속이지 않는다. MCP 비율을 footer에 표기한다.
8. disclaimer는 항상 포함한다.
```

## 11. 보고서 유형별 예시

### E 복합형 (2026-03-25 실제 출력)

```
입력: state.json (regime 🟡, divergences 2건, open questions 4건)

Phase 1 추출:
  Core Claim: "🟡 경계 (다소 완화) — 5일 군사 정지"
  Tension: 있음 (2건)   ← 대립 요소
  Timeline: 미래 분기    ← 분기 요소
  → Tension + Timeline = E 복합형

Phase 2 설계:
  분류: SPECIAL REPORT
  섹션 10개: exec + dashboard + clash + critical(1) + high(2)
             + links(4) + axis(6) + medium(2) + scenarios + unresolved

출력: reports/2026-03-25-briefing-adaptive.html
```

### C 스냅샷형 (가상)

```
입력: state.json (regime 🟢, divergences 0건, open questions 0건)

Phase 1 추출:
  Core Claim: "🟢 정상 — 전 Link 비활성"
  Tension: 없음
  Timeline: 현재 상태
  → Tension 없음 + Timeline 현재 = C 스냅샷형

Phase 2 설계:
  분류: RESEARCH NOTE
  섹션 4개: exec + dashboard + axis + observations(정리)

출력: 가장 가벼운 보고서. 대시보드 중심.
```

### A 대립형 (가상)

```
입력: divergences 3건, open questions 0건

Phase 1 추출:
  Tension: 있음 (3건)
  Timeline: 현재 상태 (미래 분기 없음)
  → Tension 있음 + Timeline 현재 = A 대립형

Phase 2 설계:
  분류: SPECIAL REPORT
  섹션: exec + dashboard + clash(3건) + observations + links

출력: Clash 섹션이 중심. 시나리오 없음.
```
