# PSF 오류 기록 + 셀프검증 체크리스트

> 발견된 오류를 기록하고, 다음 실행 시 참조하여 반복을 방지한다.
> **SKILL.md Step 1(수집) 시작 전에 이 파일의 체크리스트를 반드시 확인한다.**

## 셀프검증 체크리스트 (수집 전 필수 확인)

> 아래 항목을 수집 시작 전에 훑는다. 과거 오류에서 추출한 것.
> 새 오류가 추가되면 체크리스트도 갱신한다.

```
□ macro 데이터에서 유동성 3종(WALCL, RRPONTSYD, WTREGEN)을 확인했는가?
  → ERR-001. 유동성을 빠뜨리면 계기판의 1/4이 안 보인다.

□ 모든 S/F Property에 "방향"(↑/→/↓)을 판정했는가?
  → ERR-002. 수준만으로는 긴장 심화 vs 완화를 구분 못 한다.

□ Link 트리거에 필요한 시계열을 수집했는가? (L3.5=30일, L7=3~5일)
  → ERR-003. 현재가 1개로는 "연속"이나 "변화율"을 판정 못 한다.

□ IG 스프레드(BAMLC0A0CM)를 HY와 함께 수집했는가?
  → ERR-004. HY만 보면 진짜 위기의 선행 신호를 놓친다.

□ S4 PMI를 Tavily로 수집했으면 FRED 보조(MANEMP, INDPRO, RSAFS)로 교차검증했는가?
  → ERR-005. Tavily 단독은 정확도 보장 안 됨.

□ 인플레 기대(T5YIE)를 P1과 S1 양쪽에서 참조했는가?
  → ERR-006. P1-S1 교량 지표 누락 시 금리 판정이 한쪽만 봄.

□ F1에서 DXY만이 아니라 금(GC=F)과 TLT도 수집했는가?
  → ERR-007. 3자 교차 없이는 Divergence 패턴을 잡을 수 없다.

□ state.json에 source 태그([MCP]/[기사]/[추정])를 모든 Property에 달았는가?
  → ERR-008. 소스 없는 수치는 검증 불가.

□ quality 필드의 mcp_ratio를 계산했는가? 50% 미만이면 "품질: 낮음" 태그?
  → ERR-009. 품질 추적 없이 갱신하면 추정 기반 판정이 누적됨.

□ 활성 판 조류의 자기강화 루프가 여전히 작동하는가? (loop_active 확인)
  → F-14 방지. 루프가 꺾였으면 판 해제 검토.

□ 이벤트가 2개+ 판에 걸치면 교차를 점검했는가?
  → F-14. 교차 누락은 메가트렌드 수준의 시그널을 놓치는 것.

□ 판 조류 전수 점검했는가? (현재 활성 5개 + 후보 1개)
  → F-01. 하나라도 빠뜨리면 전체 관측이 편향됨.
```

## 오류 기록

### ERR-001: 유동성 3종 세트 누락 (2026-03-22 발견)

```
발견 상황: collection.md 최초 작성 시 WALCL, RRPONTSYD, WTREGEN 미포함.
           유동성 판정 자체가 불가능한 상태로 배포됨.

원인: ontology.md에 "순유동성 = WALCL-RRP-TGA" 공식이 있었으나
      collection.md 작성 시 ontology를 참조하지 않고 기억에 의존.

영향: 유동성은 S1(실질금리), F2(MMF), Link 전반에 영향.
      빠지면 계기판의 1/4이 안 보임.

재발 방지: collection.md 수정 시 ontology.md와 1:1 대조 필수.
          "ontology에 있는데 collection에 없는 것"을 스캔.

상태: 해결됨 (2026-03-22, 유동성 섹션 신설)
```

### ERR-002: 판정 기준에 "방향" 누락 (2026-03-22 발견)

```
발견 상황: S1 판정이 "DFII10 1.5~2.5% = 긴장"으로만 되어 있어,
           올라가는 1.88%와 내려가는 1.88%를 구분 못 함.

원인: 판정 기준을 "수준(절대값)" 기준으로만 설계.
      thinking-engine ① 레짐 판정기는 "변화율(2차 미분)"을 핵심으로 보는데,
      PSF collection에는 이 원칙이 반영 안 됨.

영향: 같은 수치에서 심화 vs 완화를 구분 못 함. 판정이 둔감해짐.

재발 방지: 모든 판정 기준에 "방향 보정" 규칙 포함.
          3개월 시계열 수집을 수준 수집과 동일 우선순위로 취급.

상태: 해결됨 (2026-03-22, 방향 보정 규칙 추가)
```

### ERR-003: Link 트리거에 시계열 기간 미명시 (2026-03-22 발견)

```
발견 상황: L3.5 "Brent 30일 >30%"인데 수집 명세에는 "Brent 일간 현재가"만.
           현재가 1개로는 30일 변화율을 계산할 수 없음.

원인: Property 수집(현재값)과 Link 점검(시계열)의 요구사항이 다른데
      수집 명세에서 이를 구분하지 않음.

영향: Link 활성/비활성 판정이 불가능하거나 추정에 의존.

재발 방지: Link 트리거 섹션에 "필요 시계열 기간" 열 추가.
          수집 프로토콜에서 "현재값 수집"과 "시계열 수집"을 분리.

상태: 해결됨 (2026-03-22, 시계열 기간 명시)
```

### ERR-004: IG 스프레드 누락 (2026-03-22 발견)

```
발견 상황: S2에 HY OAS만 있고 IG OAS(BAMLC0A0CM) 없음.
           FRED에서 검색하면 바로 나오는데 누락됨.

원인: HY가 "대표 스프레드"라는 선입견. IG는 "안전하니까 안 봐도 된다"는 암묵적 가정.

재발 방지: 신용 지표는 "HY + IG" 쌍으로 수집. 한쪽만 보지 않는다.

상태: 해결됨 (2026-03-22, BAMLC0A0CM 추가)
```

상태: 해결됨 (2026-03-22, MANEMP+INDPRO+RSAFS 보조 추가)

## 오류 기록 규칙

```
1. 오류 발견 즉시 이 파일에 기록한다. 나중에 몰아서 적지 않는다.
2. 형식: ERR-NNN, 발견 상황, 원인, 영향, 재발 방지, 상태.
3. "재발 방지"가 가장 중요한 필드. 이것이 셀프검증 체크리스트에 반영된다.
4. 해결된 오류도 삭제하지 않는다. 패턴이 보이려면 이력이 필요하다.
5. 10건 축적 후: 오류 유형별 패턴 분석. "같은 종류의 실수가 반복되는가?"
6. 셀프검증 체크리스트는 오류 기록에서 자동 파생된다. 오류가 추가되면 체크리스트도 추가.
```

## 오류 유형 분류 (패턴 추적용)

| 유형 | 설명 | 해당 오류 | 빈도 |
|---|---|---|---|
| **누락** | 있어야 할 것이 빠짐 | ERR-001, 004, 006, 007 | 4/9 (44%) ★ |
| **불완전** | 있긴 한데 부족함 | ERR-002, 003, 005 | 3/9 (33%) |
| **메타 부재** | 데이터는 있는데 품질 추적이 없음 | ERR-008, 009 | 2/9 (22%) |

**패턴:** "누락"이 가장 많다. → 근본 원인: **기억에 의존하고 원본(ontology, TE models)과 대조하지 않았다.**
→ 시스템적 해결: **collection.md 작성/수정 시 ontology.md + TE models/와 1:1 대조를 의무화.**

## 사고 오류 방어 체크리스트 (R-01~R-08)

> 데이터 오류(ERR)가 아닌 판단 오류(R)를 방어한다.
> axis-map.md 6단계(전수 점검)에서 ERR 다음에 R을 체크한다.
> 새로운 사고 오류 발견 시 추가한다. 삭제하지 않는다.

```
□ R-01 축 과신: "축 건재"라고 판단했을 때 PSF 국면도 확인했는가?
  → 축 건재 ≠ PSF 안전. 정합 매트릭스 확인 필수.
  → ⚡역설 구간(축 가속 + PSF 위기)에서도 단기 손실 가능.

□ R-02 축 의심 과잉: "축이 위험하다"고 판단했을 때 3대 조건 중 구체적으로 어느 것이 약화됐는가?
  → 구체적 조건 특정 불가 시 판(P)의 문제일 가능성.
  → 판 역풍을 축 훼손으로 오인하지 않는다.

□ R-03 루프 건너뛰기: 주간/월간 루프를 실행했는가?
  → 주간은 15분. 신호 누적 없이 분기 진단은 허공에 뜬다.

□ R-04 신호 편향: 감속·훼손 신호를 가속 신호보다 먼저 점검했는가?
  → 보고서에서 가속이 먼저 나왔으면 순서 재검토.
  → 확인 편향은 가장 흔한 사고 오류.

□ R-05 KC 방치: 활성 KC를 추적하고 있는가?
  → next_questions에 KC 관련 항목이 포함되어 있는가.
  → 설정만 하고 추적 안 하면 KC의 의미가 없다.

□ R-06 질문 방치: 직전 주간 질문에 답했는가?
  → accumulation.weekly.questions_open 확인.
  → 질문을 만들기만 하고 답을 구하지 않으면 관성.

□ R-07 레짐 무시: ⑨재정 지배를 점검했는가?
  → "거시적 배경"으로 치부하고 일상 점검에서 빠뜨리지 않았는가.
  → ⑨가 모든 축의 작동 환경을 바꾼다.

□ R-08 배관 고정: "병목은 여기"를 갱신 없이 반복하고 있는가?
  → 월간 M Loop에서 배관 변화 확인.
  → 병목 이동 = 가치 포착점 이동.
```
