# psf-monitor 작동 원리

> 세계의 현재 상태를 읽고 → 3층으로 전파를 추적하고 → 축까지의 거리를 측정하고 → 위험을 경보한다
>
> 이 문서는 `LLM기반 시스템 표준 아키텍처`를 psf-monitor에 적용한 것이다.

---

## 1. 시스템 5요소 (Five Elements)

```
┌─────────────────────────────────────────────────────────────────┐
│                                                                 │
│                        ┌───────────┐                            │
│                        │  두뇌(LLM) │                            │
│                        │  Claude   │                            │
│                        └─────┬─────┘                            │
│                     읽기↑    │판단    ↑읽기                      │
│                ┌─────────┐   │   ┌─────────┐                    │
│                │ 지침(MD) │   │   │기억(JSON)│                    │
│                │ 9개 문서 │   │   │state +  │                    │
│                │ 7개 원칙 │   │   │history  │                    │
│                └─────────┘   │   └─────────┘                    │
│                              ↓                                  │
│                ┌─────────────────────────┐                      │
│                │      손발 (MCP/API)      │                      │
│                │  macro + Tavily +        │                      │
│                │  DeFiLlama + CoinGecko   │                      │
│                └─────────────┬───────────┘                      │
│                              │                                  │
│                        ┌─────┴─────┐                            │
│                        │ 뼈대(코드) │                            │
│                        │  (없음)   │                            │
│                        └───────────┘                            │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 요소 정의

```
┌────────┬──────────┬──────────────────────────────────────────────┐
│ 요소    │ 비유      │ psf-monitor에서의 역할                        │
├────────┼──────────┼──────────────────────────────────────────────┤
│ 뼈대    │ 근골격계  │ 없음. 순수 프롬프트 기반 에이전트.             │
│ (Code) │          │ 코드가 할 일(검증, 렌더링)을 LLM이 직접 수행.  │
│        │          │ ★ 5요소 곱셈 법칙의 예외:                     │
│        │          │   뼈대 = 0이지만 LLM이 대행하므로 시스템 작동.  │
│        │          │   단, 일관성은 지침(MD)이 보장해야 함.          │
├────────┼──────────┼──────────────────────────────────────────────┤
│ 지침    │ 교범      │ CLAUDE.md (정체성+7개 원칙),                  │
│ (MD)   │          │ SKILL.md (Phase 0~출력),                     │
│        │          │ ontology.md (운용 헌법),                      │
│        │          │ axis.md (축 정의서),                          │
│        │          │ axis-map.md (발견 프로토콜 6단계),             │
│        │          │ monitoring.md (D/W/M/Q 루프),                 │
│        │          │ collection.md (매핑 명세),                    │
│        │          │ projection-engine.md (12개월 투영),            │
│        │          │ errors.md (오류 기록+체크리스트)                │
│        │          │ → 코드 부재를 보완. 시스템 진화의 100%가 여기.  │
├────────┼──────────┼──────────────────────────────────────────────┤
│ 두뇌    │ 뇌        │ Claude. 지침을 읽고 관측하고 기록한다.         │
│ (LLM)  │          │ 매 세션 백지. 기억(JSON)으로 연속성 보완.      │
│        │          │ 코드 부재로 뼈대 역할까지 겸함.                │
├────────┼──────────┼──────────────────────────────────────────────┤
│ 기억    │ 해마/기록 │ state.json (현재 관측값),                     │
│(Memory)│          │ projection.json (12개월 시나리오),              │
│        │          │ history/*.json (관측 이력 스냅샷),              │
│        │          │ reports/ (HTML 보고서)                         │
│        │          │ → 세션 간 브릿지. 어제의 관측을 기억.           │
├────────┼──────────┼──────────────────────────────────────────────┤
│ 손발    │ 손, 발, 입│ macro/indicators/latest.json (정량 읽기),     │
│ (Tools)│          │ Tavily (P층 정성 뉴스),                       │
│        │          │ DeFiLlama, CoinGecko (축 신호),               │
│        │          │ CoinMetrics, FRED (축 신호 보충)               │
│        │          │ → macro가 유일한 정량 수집자.                  │
│        │          │   P층 정성+축 신호만 직접 MCP 동원.            │
└────────┴──────────┴──────────────────────────────────────────────┘
```

### 요소 간 관계 법칙 (psf-monitor 적용)

```
법칙 1: 두뇌는 지침 없이 일반론만 낸다.
        → 9개 MD 문서가 "관측하되 해석하지 않는" 품질을 결정.

법칙 2: 두뇌는 기억 없이 매번 맨땅이다.
        → state.json이 "뭐가 변했는가"를 비교할 기준선 제공.

법칙 3: 두뇌는 손발 없이 추측만 한다.
        → macro + MCP가 데이터 사실 기반 제공. [MCP] 50% 미만이면 품질 낮음.

법칙 4: 뼈대는 나머지 4요소를 연결한다.
        → psf-monitor는 뼈대 없음. 지침(MD)이 연결 역할 대행.
          collection.md가 macro→PSF 연결, interfaces-macro.md가 시스템 간 연결.

법칙 5: 5요소는 곱셈 관계다.
        → 뼈대 = 0이지만 LLM이 대행 → 실질적으로 4요소 곱셈.
          지침 품질이 전체 성능의 병목. (코드가 강제하지 못하므로)
```

## 2. 계층 구조 (Layer Architecture)

```
┌─────────────────────────────────────────────────────────────┐
│ Layer 5   자율 계층 (Autonomy Layer)                         │
│           Phase 0 자동 트리거 (macro stale 시 수집 대행)      │
│           D/W/M/Q 4중 루프 자동 리듬                          │
│           울타리: 관측=무승인, 판정수정=승인, 투자판단=금지     │
├─────────────────────────────────────────────────────────────┤
│ Layer 4   진화 계층 (Evolution Layer)                        │
│           errors.md 셀프검증 → 오류 패턴 누적 → 방어규칙 추가│
│           Q Loop에서 ontology.md / axis.md 개정 가능          │
│           인지 오류 방어 R-01~R-08                            │
├─────────────────────────────────────────────────────────────┤
│ Layer 3   실행 계층 (Execution Layer)                        │
│           트리거 → 데이터 수집 → axis-map 6단계 → 판정 → 출력│
│           D Loop(일일), W Loop(주간), M Loop(월간), Q Loop(분기)│
├─────────────────────────────────────────────────────────────┤
│ Layer 2   기억 계층 (Memory Layer)                           │
│           단기: state.json + projection.json                 │
│           중기: history/ (일일/주간/월간 스냅샷)               │
│           장기: (Notion 아카이브 — 미구현)                     │
├─────────────────────────────────────────────────────────────┤
│ Layer 1   인프라 계층 (Infrastructure Layer)                  │
│           코드 없음. MCP 연결(macro, Tavily, DeFiLlama 등).   │
│           파일시스템(JSON + MD). 스케줄러 없음.                │
└─────────────────────────────────────────────────────────────┘
```

### 계층 간 의존 관계

```
Layer 5 (자율) → Layer 4 (진화) → Layer 3 (실행) → Layer 2 (기억) → Layer 1 (인프라)

위 계층은 아래 계층 없이 작동 불가.
아래 계층은 위 계층 없이도 작동 가능.

psf-monitor 현재 구성:
  Layer 1: ✅ (MCP 연결 + 파일시스템. 코드/스케줄러 없음)
  Layer 2: ✅ (state.json + history/ + projection.json)
  Layer 3: ✅ (D/W/M/Q 루프 + axis-map 6단계)
  Layer 4: ✅ (errors.md + Q Loop 온톨로지 개정)
  Layer 5: ✅ (Phase 0 자동 트리거 + 4중 루프 리듬)
  → 완전 구성 (전 계층)
```

## 3. 세션 생명주기 (Data Flow)

```
━━━ 세션 시작 ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Phase A: 맥락 복원
├── state.json → LLM에 로딩 (현재 3층 상태 + 축 상태)
├── history/ → 최근 스냅샷 delta만 로딩 ("뭐가 변했는가")
├── CLAUDE.md → LLM에 로딩 (정체성 + 7개 원칙)
├── errors.md → 셀프검증 체크리스트 확인 (12항목)
└── → "어제의 나"를 복원

Phase B: 상황 스캔
├── macro/indicators/latest.json → 신선도 확인
│   fresh(~3일) → 사용
│   stale(3~7일) / expired(7일+) → macro 수집 절차 자동 대행
├── state.json → 이전 미해소 질문 확인
├── monitoring.md → 오늘의 루프 유형 결정 (D/W/M/Q)
└── → "오늘 할 일" 결정

Phase C: 스킬 실행
├── SKILL.md + axis-map.md 로딩 → 실행 프로토콜
├── 데이터 수집 (macro 읽기 + Tavily + 맥락 MCP)
├── axis-map 6단계 (변화→인과→축→이상→위험→전수)
├── 판정 (PSF 국면 + 축 상태 + Link 활성)
├── 출력 생성 (텍스트 브리핑 + HTML 보고서)
└── → 산출물 생성

Phase D: 후처리
├── state.json 갱신 (현재 상태 덮어쓰기)
├── history/YYYY-MM-DD.json 추가 (오늘 스냅샷)
├── accumulation 누적 갱신 (주간/월간)
├── reports/ HTML 저장
├── PM_핵심예측 추출 (수치 범위 + 기한 필수)
└── projection.json 조건 점검 (분기 조건 충족 시 확률 조정)

━━━ 세션 종료 ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

LLM(두뇌) → 소멸
기억(JSON) → 영속 ← 세션 간 브릿지
지침(MD)   → 영속 (Q Loop 리뷰 시 갱신)
```

### 세션 간 연결

```
세션 A                    memory/                   세션 B
─────────                ─────────                 ─────────
판단 실행                                           맥락 복원
  ↓                                                   ↑
결과 저장 ──→ state.json ─────────────────→ state.json 로딩
스냅샷 저장 ──→ history/ ──────────────────→ delta 비교
투영 갱신 ──→ projection.json ─────────────→ 시나리오 참조
보고서 저장 ──→ reports/ ──────────────────→ 이전 브리핑 참조

대화 히스토리 → 소멸 (세션 내부에만 존재)
판단 결과 → 영속 (state.json + history/에 저장)
```

## 4. 기억 계층 설계 (Memory Architecture)

### 3층 기억 구조

```
┌──────────────────────────────────────────────────────────────┐
│ 층      │ 저장소          │ 접근 속도 │ 보존 기간 │ 용도       │
├─────────┼────────────────┼──────────┼──────────┼────────────┤
│ 단기기억 │ state.json      │ 즉시     │ 항상 최신 │ 현재 3층   │
│         │ projection.json │          │ 1건 유지  │ 현재 투영  │
│         │                │          │          │ 세션 시작  │
│         │                │          │          │ 즉시 로딩  │
├─────────┼────────────────┼──────────┼──────────┼────────────┤
│ 중기기억 │ history/        │ 빠름     │ 영구     │ 시계열 비교│
│         │  YYYY-MM-DD.json│         │          │ 일일 스냅샷│
│         │  YYYY-W##.json  │          │          │ 주간 요약  │
│         │  YYYY-MM.json   │          │          │ 월간 요약  │
│         │ reports/        │          │          │ HTML 브리핑│
├─────────┼────────────────┼──────────┼──────────┼────────────┤
│ 장기기억 │ (미구현)        │ —        │ —        │ Notion 등  │
│         │                │          │          │ 전체 아카이브│
└─────────┴────────────────┴──────────┴──────────┴────────────┘
```

### 기억 간 동기화

```
단기(state/) ──스냅샷 생성──→ 중기(history/)
중기(history/) ──아카이브──→ 장기(미구현)

동기화 시점:
├── 단기→중기: 매 스킬 실행 완료 시 (자동)
├── 중기 누적: W Loop에서 주간 요약, M Loop에서 월간 요약
└── 장기: 미구현 (향후 Notion 연동 가능)
```

기억이 없으면:
  매 세션이 처음부터 시작. 어제의 Link 상태를 모름.
  "뭐가 변했는가?"를 비교할 기준이 없음.

기억이 있으면:
  "어제 L3 활성 → 오늘 L3.5까지 확산" 추적 가능.
  "2주 전 BEI 2.34% → 오늘 2.63% (+29bp)" 변화율 계산 가능.

## 5. 지침 계층 설계 (Instruction Architecture)

### 지침의 계층

```
┌──────────────┬──────────────────────┬────────────────────────────┐
│ 층            │ 파일                  │ psf-monitor 역할            │
├──────────────┼──────────────────────┼────────────────────────────┤
│ L0 시스템     │ 시스템 프롬프트        │ LLM 기본 행동 규칙         │
│ (불변)       │ (Anthropic 제공)      │ 통제 불가                   │
├──────────────┼──────────────────────┼────────────────────────────┤
│ L1 오케스트라 │ CLAUDE.md             │ 정체성, 7개 불변 원칙,      │
│ (드물게 변경) │                      │ 위치와 관계, 참조 파일 계층  │
├──────────────┼──────────────────────┼────────────────────────────┤
│ L2 스킬      │ SKILL.md              │ Phase 0~출력, D/W/M/Q 루프 │
│ (자주 변경)  │ ontology.md           │ 운용 헌법 (P/S/F, Link)    │
│              │ axis.md               │ 축 정의서 (9대, 3조건)     │
│              │ axis-map.md           │ 발견 프로토콜 6단계         │
│              │ monitoring.md         │ 모니터링 루틴              │
│              │ collection.md         │ macro→PSF 매핑 명세        │
│              │ projection-engine.md  │ 12개월 투영 프로토콜        │
│              │ errors.md             │ 오류 기록 + 셀프검증        │
├──────────────┼──────────────────────┼────────────────────────────┤
│ L3 레퍼런스  │ 원본-PSF 온톨로지      │ 최고 우선순위 원본          │
│ (필요 시 변경)│ 엔진.md              │                            │
│              │ interfaces-macro.md   │ macro ↔ psf-monitor 접점   │
└──────────────┴──────────────────────┴────────────────────────────┘
```

### 지침 로딩 순서

```
세션 시작 시:
L0 (시스템 프롬프트) → 자동 로딩 (항상)
L1 (CLAUDE.md) → 자동 로딩 (항상)

스킬 트리거 시:
L2 (SKILL.md + 해당 루프) → 트리거 시 로딩
L3 (원본, interfaces) → 충돌 시 또는 필요 시 로딩

원칙: L0~L1은 항상, L2는 해당 루프만, L3는 필요한 것만.

문서 우선 순위:
  원본 > ontology > axis > collection > axis-map
```

### 지침 품질 표준 (7단계 프로토콜 적용)

```
psf-monitor 스킬의 7단계 충족 현황:

Step 1 WHY:    ✅ "세계의 현재 상태를 관측하고 축까지의 거리를 측정"
Step 2 WHAT:   ✅ 관측(IN), 해석/예측/행동권고(OUT), CE/TE/macro(ADJACENT)
Step 3 WHEN:   ✅ D/W/M/Q 트리거 + 긴급 + 수시
Step 4 HOW:    ✅ Phase 0 → 수집 → axis-map 6단계 → 기록 → 출력
Step 5 FORMAT: ✅ 텍스트 브리핑(20줄) + HTML + state.json
Step 6 FAIL:   ✅ F-01~F-12 테이블 + 오류 유형 패턴(누락42%/불완전33%/경계25%)
Step 7 EVOLVE: ✅ Self-Audit Q1~Q5 + 갱신 주기(D/W/Q) + 불변 규칙 8개

→ Step 1~7 완전 충족. 점수 7/7. 정밀도 90점+ (자기 진화 가능)
```

## 6. 도구 계층 설계 (Tool Architecture)

### 도구 분류

```
┌────────────┬──────────────────────────┬──────────────────────────┐
│ 분류        │ 역할                     │ psf-monitor 구현           │
├────────────┼──────────────────────────┼──────────────────────────┤
│ 눈 (감각)   │ 외부 세계 관찰           │ Tavily (P층 정성 뉴스)    │
│            │                          │ WebSearch (축 신호 탐색)  │
├────────────┼──────────────────────────┼──────────────────────────┤
│ 손 (조작)   │ 데이터 수집 + 가공       │ FRED (macro 경유),        │

│            │                          │ latest.json (파일 읽기)   │
├────────────┼──────────────────────────┼──────────────────────────┤

│            │                          │ projection.json, reports/ │
└────────────┴──────────────────────────┴──────────────────────────┘
```

### 도구 운영 원칙

```
1. macro 우선 + MCP 보충
├── 정량 46개: macro/latest.json이 1차 (유일)
├── P층 정성: Tavily가 1차, WebSearch가 폴백
├── 축 신호: 질문이 소스를 결정 (고정 목록 아님)
└── 수집 실패 시: "미확인 — [사유]" 표기 후 진행

2. 맥락 기반 동원
├── 모든 MCP를 매번 쓰지 않음
├── "오늘의 데이터에서 답이 필요한 질문"이 소스를 결정
└── 변화 없으면 기본 소스(macro + Tavily)로 충분

3. 컨텍스트 비용 인식
├── macro latest.json 전체 로딩 (46개 → 필수)
├── 축 신호 MCP는 필요한 것만
└── 결과에서 필요한 부분만 추출, 원문 폐기
```

## 7. 자율 계층 설계 (Autonomy Architecture)

### 자율 3조건

```
자율 = 기억(Memory) + 루프(Loop) + 울타리(Guardrail)

기억: state.json이 어제 상태를 알려줘야 오늘 "변화"를 감지 가능
루프: D→W→M→Q 4중 루프가 관측→판정→기록→검증의 순환 보장
울타리: 관측=무승인, 구조 수정=승인, 투자 판단=금지

하나라도 빠지면:
├── 기억 없는 자율 = 매번 초기화 ("뭐가 변했는가?" 불가)
├── 루프 없는 자율 = 1회성 스냅샷 (축 추적 불가)
└── 울타리 없는 자율 = 관측자가 해석자로 변질
```

### 자율 루프 상세

```
감지 (Sense)
├── 세션 시작: state.json 로딩 → 이전 상태 확인
├── macro 신선도: latest.json date 확인 → stale 시 자동 대행
├── 이전 미해소: next_questions 확인
└── 루프 유형: monitoring.md → D/W/M/Q 판별
    ↓
판단 (Decide)
├── axis-map 6단계 실행 (변화→인과→축→이상→위험→전수)
├── PSF 국면 판정 (🟢/🟡/🔴)
├── 축 상태 판정 (건재/감속/훼손)
└── 감속 신호를 가속 신호보다 먼저 점검 (확인 편향 방어)
    ↓
실행 (Act)
├── state.json 갱신
├── history/ 스냅샷 저장
├── 보고서 출력 (텍스트 + HTML)
└── PM_핵심예측 추출
    ↓
검증 (Verify)
├── errors.md 셀프검증 체크리스트 (12항목)
├── axis-map 6단계 전수 점검 ("놓친 것 없는가?")
├── 오류 발견 → errors.md에 기록
└── 시스템성 오류 → 방어 규칙 추가
    ↓
기억 갱신 (Remember)
├── state.json 갱신
├── history/ 스냅샷 추가
├── accumulation 누적 갱신
└── projection.json 조건 점검
    ↓
다음 세션 → 감지 (루프)
```

### 자율 수준 (psf-monitor 현재 위치)

```
Level 0 — 수동:         기억 없음. 매번 맨땅.
Level 1 — 자율 감지:     ✅ state.json 로딩 + 신선도 체크 + 루프 유형 판별
Level 2 — 자율 판단:     ✅ axis-map 6단계 자율 실행 + macro 자동 대행 제안
Level 3 — 자율 실행:     ✅ Phase 0 자동 트리거 + D/W/M/Q 루프 자동 리듬

psf-monitor → Level 3 (자율 실행) ← 현재 위치
  울타리 안에서 관측·판정·기록을 무승인 실행.
  온톨로지 개정, 투영 전면 재구성은 승인 필요 (Q Loop).
```

### 안전 울타리 (Guardrail)

```
Green Zone (무승인 허용):
├── state.json 갱신, history/ 스냅샷 저장
├── 데이터 수집 (macro 읽기, MCP 호출)
├── PSF 국면 판정, 축 상태 판정
├── 보고서 생성 (텍스트 + HTML)
├── macro stale 시 수집 절차 자동 대행
├── accumulation 누적 갱신
└── errors.md 오류 기록

Yellow Zone (승인 필요):
├── ontology.md / axis.md 개정 (Q Loop)
├── 투영 전면 재구성 (projection.json 리셋)
├── 축 후보 승격/강등
├── 새 Link 정의 추가
└── 인터페이스 변경 (interfaces-macro.md)

Red Zone (절대 금지):
├── 투자 판단 ("사야/팔아야")
├── 인과 해석 ("왜 그런지" — CE/TE의 일)
├── 예측 ("앞으로 어떻게 될지" — CE/TE의 일)
├── 파일/데이터 영구 삭제
└── 울타리 자체 수정
```

## 8. 진화 설계 (Evolution Architecture)

### 진화가 일어나는 곳

```
코드(뼈대)   → 없음. 변경 대상 없음.
지침(MD)     → 시스템 진화의 100%.  ← 진화의 유일한 경로
LLM(두뇌)   → 외부 제공. 통제 불가.
기억(JSON)   → 매일 자동 누적.
도구(MCP)    → 필요 시 추가/교체.
```

### 진화 루프

```
운영 (매일 D Loop)
    ↓
셀프검증 (매 실행)
├── errors.md 체크리스트 12항목
├── 인지 오류 방어 R-01~R-08
    ↓
오류 감지 (자동)
├── 일회성 → state.json에 "[주의]" 태그
└── 시스템성 → errors.md에 ERR-xxx 추가
    ↓
패턴 누적 (주간 W Loop)
├── 동일 오류 반복 → severity 승격
├── 오류 유형별 집계 (누락 44%, 불완전 33%, 메타 부재 22%)
└── 핵심 대책: ontology ↔ collection 1:1 교차 검증
    ↓
구조 리뷰 (분기 Q Loop)
├── 축 건강 진단 (3조건 재검증)
├── ontology.md / axis.md 개정 가능
├── 후보(⑤⑦) 승격 조건 점검
├── [미분류] 패턴이 새 구조를 형성하는가?
└── 투영 전면 재구성
    ↓
다음 운영에 반영 → 루프
```

### 진화의 불변 규칙

```
1. 5요소 구조 자체는 바꾸지 않는다. (코드 부재도 설계 선택)
2. 기억의 3층 구조(단기/중기/장기)는 유지한다.
3. errors.md에서 "불편한 질문"을 제거하지 않는다.
4. 오류 기록(ERR-xxx)은 삭제하지 않고 누적만 한다.
5. 울타리의 Red Zone을 축소하지 않는다.
6. 관측과 해석의 분리를 훼손하지 않는다.
7. 원본-PSF 온톨로지 엔진.md의 최고 우선순위를 바꾸지 않는다.
```
