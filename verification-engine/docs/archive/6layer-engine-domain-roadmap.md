## 1. 현재 엔진의 실제 구조

현재 엔진은 **"범용 뼈대 위에 금융·투자 전문 규칙이 탑재된 상태"**다.
두 층위가 명확히 분리되어 있으며, 이 구조 자체가 향후 확장의 토대가 된다.

```
6-Layer Verification Engine 구조
│
├── Layer 1: 범용 뼈대 (도메인 무관 — 현재 완성)
│   ├── 6층 파이프라인: Fact / Norm / Logic / Temporal / Incentive / Omission
│   ├── 불변 원칙 5개 ("LLM은 비교자이지 판단자가 아니다" 등)
│   ├── Phase 0~4 실행 흐름 (문서 입력 → 판정 → JSON 저장)
│   ├── 판정 체계: 🟢🟡🔴⚫ 4등급
│   └── → 어떤 도메인의 문서든 이 뼈대로 처리 가능
│
└── Layer 2: 도메인 전문 규칙 (현재 금융·투자 편향)
    ├── NORM 체크리스트: equity / crypto / finance_alt / legal_contract
    ├── RULES 논리 규칙: 주식(5) / 크립토(2) / 법률(3) / 금융(6) / 공통(1) = 17개
    └── OMISSION 체크리스트: 반도체 / 크립토 / 바이오 / 에너지 / 부동산 / 금융대체 / 법률
```

## 2. 도메인별 현재 작동 수준

| 도메인 | Layer 1 뼈대 | Layer 2 규칙 | 실제 판정 품질 | 비고 |
|---|---|---|---|---|
| 금융·투자 리포트 | ✅ | ✅ 풍부 | 🟢 높음 | 핵심 도메인 |
| 크립토 리서치 | ✅ | ✅ 있음 | 🟢 높음 | 핵심 도메인 |
| 법률 계약서 | ✅ | ✅ 있음 | 🟡 중간 | 기본 커버 |
| 매크로 리포트 | ✅ | 🔴 없음 → 추가 예정 | ⚫ 현재 판정 불가 | L2 작업 대상 |
| 지정학 분석 | ✅ | 🔴 없음 → 추가 예정 | ⚫ 현재 판정 불가 | L3 작업 대상 |
| SaaS/IT 리포트 | ✅ | 🔴 없음 → 추가 예정 | ⚫ 현재 판정 불가 | L3 작업 대상 |
| **의료 리포트** | ✅ | ❌ 미구축 | ⚫ 판정 불가 | 향후 확장 |
| **법률 판례·의견서** | ✅ | ❌ 미구축 | ⚫ 판정 불가 | 향후 확장 |
| **학술 논문** | ✅ | ❌ 미구축 | ⚫ 판정 불가 | 향후 확장 |
| **정책·규제 문서** | ✅ | ❌ 미구축 | ⚫ 판정 불가 | 향후 확장 |
| **부동산 실물 리포트** | ✅ | ❌ 미구축 | ⚫ 판정 불가 | 향후 확장 |

> **판정 불가의 의미:** 6층 파이프라인은 실행되지만, NORM·RULES·OMISSION에
> 해당 도메인 규칙이 없어 ① Fact·② Norm·③ Logic·⑥ Omission이 ⚫ NO BASIS로 출력됨.
> 뼈대가 작동할 뿐, 실질적 판정 깊이가 없는 상태.

## 4. 향후 확장 도메인별 필요 작업

### 4-1. 의료 리포트 (Clinical Research / Medical Report)

**우선순위:** 🟡 중간 (희발님 주요 사용 도메인 아님, 수익화 고려 시 검토)

**필요한 작업:**

| 구성 요소 | 내용 | 난이도 |
|---|---|---|
| NORM 추가 | CONSORT 기준, IRB 승인 공시, 이해충돌 선언 (ICMJE) | 중 |
| RULES 추가 | 샘플 수 미기재, p-value 기준 미명시, 비교군 없는 단일군 연구, 효과크기 미보고 | 중 |
| OMISSION 추가 | 부작용 보고, 대조군 설계, 추적 기간, 탈락률 | 중 |
| MCP 추가 | PubMed API, ClinicalTrials.gov, FDA 승인 DB | 높음 |
| 1차 소스 기준 | 피어리뷰 논문, FDA/식약처 공식 문서 | - |

**확장 시 판정 가능 문서:**
- 임상시험 결과 리포트
- 제약사 파이프라인 분석 (바이오 섹터 투자 연계 가능)
- 의료기기 허가 신청서

---

### 4-2. 법률 판례·의견서 (Legal Opinion / Case Law)

**우선순위:** 🟡 중간 (현재 계약서 커버, 판례·의견서는 미구축)

**필요한 작업:**

| 구성 요소 | 내용 | 난이도 |
|---|---|---|
| NORM 추가 | 판례 인용 형식, 관할 법원 명시, 법령 조문 인용 규칙 | 낮음 |
| RULES 추가 | 인용 판례의 현행 유효성, 법령 개정 여부, 반대 판례 무시 | 중 |
| OMISSION 추가 | 반대 판례, 소수의견, 법령 개정 가능성, 상급심 가능성 | 중 |
| MCP 추가 | 국가법령정보센터, 대법원 판례 DB, 헌법재판소 결정문 | 높음 |
| 1차 소스 기준 | 대법원 판결문, 국가법령정보센터 조문 | - |

**확장 시 판정 가능 문서:**
- 법률 자문 의견서
- 규제 리스크 분석 (투자 리포트 내 법률 섹션 연계 가능)
- 판례 기반 분쟁 리스크 평가

### 4-3. 학술 논문 (Academic Paper)

**우선순위:** 🟡 중간 (PSF 프레임워크 근거 검증, 지정학 역사적 사례 확인에 유용)

**필요한 작업:**

| 구성 요소 | 내용 | 난이도 |
|---|---|---|
| NORM 추가 | 방법론 섹션 필수, 데이터 공개 여부, 피인용 검증, 저널 등급 | 낮음 |
| RULES 추가 | 재현 가능성 미기재, 이해충돌 미선언, 단일 데이터셋 의존 | 중 |
| OMISSION 추가 | 한계 섹션, 대안 해석, 외적 타당도, 선택 편향 | 중 |
| MCP 추가 | Semantic Scholar, arXiv, SSRN | 중 |
| 1차 소스 기준 | DOI 원문, 피인용 논문, 데이터 저장소 | - |

**확장 시 판정 가능 문서:**
- 매크로 경제 연구 논문
- 지정학 리스크 학술 분석

## 5. 확장 우선순위 판단 기준

```
우선순위 결정 트리
│
├── Q1: 희발님의 투자 의사결정에 직접 연결되는가?
│   ├── YES → 최우선 (매크로·지정학·SaaS = 현재 스프린트)
│   └── NO  → Q2로
│
├── Q2: 수익화 리서치 서비스 대상 독자에게 가치가 있는가?
│   ├── YES → 중기 확장 (의료·법률 = Phase 2)
│   └── NO  → Q3로
│
└── Q3: 기존 도메인 검증을 보강하는 역할인가?
    ├── YES → 장기 추가 (학술·정책 = Phase 3)
    └── NO  → 현재 로드맵 외
```

## 7. 규칙 엔진 유지보수 — 정기 업그레이드 전략

### 7-1. 왜 정기 업그레이드가 불가피한가

```
업그레이드가 필요한 3가지 이유
│
├── 시장 구조 자체가 변한다
│   ├── 2022년: LUNA 붕괴 → 알고리즘 스테이블코인 규칙 신규 필요
│   ├── 2023년: AI 테마 → SaaS Rule of 40이 무의미해진 구간 발생
│   └── 2025년: 관세전쟁 → 지정학 Omission 체크리스트 필요
│       → 시장이 바뀌면 "무엇을 검증해야 하는가"도 바뀜
│
├── 규제·법령이 바뀐다
│   ├── NORM 체크리스트의 기준은 법령·모범규준 기반
│   └── 금감원 규정 개정, SEC Rule 변경, 가상자산법 시행
│       → 면책 조항 기준, 이해충돌 공시 요건이 달라짐
│
└── 실전 실패가 쌓인다
    ├── 🟢 판정 받은 리포트가 실제로 틀리는 케이스 발생
    └── 현재 규칙이 못 잡는 패턴이 발견됨
        → 이것이 가장 강력한 업그레이드 소재
```

### 7-2. 업데이트 주기별 분류

| 주기 | 트리거 | 대상 파일 | 작업 내용 |
|---|---|---|---|
| 🔴 수시 | MCP 소스 장애·API 변경 / 명백한 판정 오류 | RULES.md | 개별 규칙 수정·시리즈 ID 교체 |
| 🟡 월 1회 | 오답노트 리뷰 (L4-04) | RULES.md | 실패 케이스 → 신규 규칙 추가 |
| 🔵 분기 1회 | 법령 변경 확인 / 새 리스크 유형 점검 | CHECKLISTS.md | 섹터 항목 추가·갱신 |
| 🟣 연 1회 | 글로벌 표준 업데이트 (CFA, IFCN) | SKILL.md | 파이프라인 구조 재검토 |

### 7-3. 변하지 않는 것 — 건드리지 않아도 되는 것

```
불변 요소 (업데이트 대상 아님)
├── SKILL.md 불변 원칙 5개
│   "LLM은 비교자이지 판단자가 아니다" 등
│   → 도메인·시장과 무관한 설계 철학
├── 6층 파이프라인 구조 자체
│   Fact → Norm → Logic → Temporal → Incentive → Omission
└── 판정 체계 🟢🟡🔴⚫
    → 기준 없이 판정 안 한다는 원칙 자체
```

## 8. 트리거 시그널 기반 자동화 전략

### 8-1. 자동화 대상의 구분

```
"트리거 시그널 → 자동 업데이트"를 두 유형으로 반드시 분리

├── A유형: 트리거 → 자동 알림·재검증 실행  ✅ 완전 자동화 가능
└── B유형: 트리거 → 규칙 자동 수정         🔴 위험 — 반자동만 허용
```

### 8-2. A유형 — 완전 자동화 가능 (heartbeat.py 연동)

```
heartbeat.py가 감지하는 트리거 시그널
│
├── Net Liquidity 전일 대비 ±2% 초과
├── MVRV > 3.0 또는 < 1.0
├── BEI 5Y > 2.5%
├── HY 스프레드 급등
└── "Hormuz" / "FOMC" / "관세" 뉴스 감지
        ↓
현재: 트리거 → Telegram 알림만 전송
        ↓
확장: 트리거 → 검증 엔진 자동 실행
├── 관련 보유 분석 리포트 재검증 트리거
├── 해당 규칙(lr_*) 자동 대입 → 결과 Telegram 전송
└── Notion + invest-db 자동 저장

예시 흐름:
MVRV > 3.0 감지
→ 보유 크립토 리포트 자동 재검증 실행
→ lr_007_v2 규칙 자동 대입
→ "lr_007_v2: 🔴 사이클 22개월 경과, 고평가 구간 매수 주의"
→ Telegram 발송 + Notion 저장
```

### 8-3. B유형 — 규칙 자동 수정 (완전 자동화 금지 이유)

```
"실패가 쌓이면 MVRV 임계값을 시스템이 자동으로 3.0→2.8로 조정"
이 방식의 위험 지점 3개:
│
├── 위험 1: 과적합 (Overfitting)
│   ├── 최근 3개월 데이터로 규칙 자동 수정
│   ├── Bull 사이클 → MVRV 기준 계속 상향
│   └── Bear 전환 시 규칙이 완전히 무력화됨
│
├── 위험 2: 자기 검증 불가
│   ├── 시스템이 규칙을 수정하고
│   │   동일 시스템이 수정된 규칙을 검증
│   └── GIPS Verifier 원칙 위반
│       "검증자는 피검증자와 독립되어야 한다"
│
└── 위험 3: 추적 불가
    ├── 규칙이 언제 왜 바뀌었는지 기록 없이 자동 수정
    └── 판정 결과의 신뢰도 근거가 사라짐
```

### 8-4. 권장 설계 — 자동화 가능 범위와 인간 결정 범위

```
Layer 1: 완전 자동화 ✅
├── 시그널 감지 → Telegram 알림
├── 시그널 감지 → 검증 엔진 자동 재실행
├── 검증 결과 → Notion + invest-db 자동 저장
└── 실패 판정 → "이 규칙을 재검토하세요" 알림 발송

Layer 2: 반자동 (시스템 초안 + 인간 승인) 🟡
├── 실패 케이스 누적 자동 감지
│   "lr_007 규칙이 지난 30일간 3회 중 2회 틀렸습니다"
│   "수정 초안: MVRV 임계값 3.0→2.8 검토 권장"
└── 희발님 승인 후 → RULES.md 업데이트 실행

Layer 3: 인간 전용 (자동화 금지) 🔴
├── 규칙 조건 최종 수정 (판단권은 인간에게)
├── 새 도메인 추가 결정
└── 기존 규칙 삭제
```

### 8-5. heartbeat.py 추가 모듈 설계

```
기존 heartbeat.py에 추가할 2개 모듈

모듈 1: 실시간 재검증 트리거
├── 트리거 시그널 감지 시
│   verification_results 테이블에서
│   관련 도메인 마지막 검증 시점 조회
├── 검증 후 30일 이상 경과 시 → 자동 재검증 실행
└── 재검증 결과 → Telegram + Notion 저장

모듈 2: 월간 규칙 정확도 집계 (월 1회 자동 실행)

│   규칙별 정확도 = outcome_correct / 전체 적용 건수
├── 정확도 < 60% 규칙 → 재검토 알림 발송
│   예: "lr_007: 최근 30일 정확도 40% — 재검토 권장"
└── 희발님 검토 → 승인 시 RULES.md 수정 (반자동)
```

## 9. 확장 원칙

도메인을 추가할 때 반드시 지켜야 할 원칙:

```
원칙 1: 뼈대는 건드리지 않는다
  SKILL.md의 6층 파이프라인과 불변 원칙은 수정하지 않는다.
  새 도메인은 반드시 RULES.md + CHECKLISTS.md 파일에만 추가한다.

원칙 2: MCP 먼저, 규칙 나중
  1차 소스 MCP 없이 RULES를 추가하면 ⚫ NO BASIS만 늘어난다.
  MCP 구축 확인 후 해당 도메인 규칙 작성 시작.

원칙 3: 실전 실패에서 역추출
  새 도메인 규칙은 책·논문이 아니라
  실제로 검증 실패했던 케이스에서 귀납적으로 도출한다.
  월 1회 오답노트 리뷰(L4-04) 결과를 신규 도메인 규칙으로 전환.

원칙 4: 섹터보다 문서 유형 우선
  "의료 섹터"가 아니라 "임상시험 결과 리포트"처럼
  검증 대상 문서 유형 기준으로 도메인을 정의한다.
  동일 섹터라도 문서 유형에 따라 다른 규칙이 필요하다.
```
