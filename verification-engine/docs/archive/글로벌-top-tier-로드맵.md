## 현재 수준 진단

### 잘 하고 있는 것

| 영역 | 현재 수준 | 벤치마크 |
|---|---|---|
| 구조화된 6층 프레임워크 | 상위 10% | 대부분의 팩트체킹 시스템은 단일 차원(사실/거짓) |
| MCP 1차 소스 강제 조회 | 상위 5% | LLM 기반 시스템 중 1차 소스 강제는 드묾 |
| KC(Kill Condition) 3단 역추적 | 독자적 | 학술적 근거 있으나 상용 시스템에선 미구현 |
| BBJ Break 4관점 전환법 | 독자적 | ACH(경쟁가설분석)를 투자 검증에 적용한 사례 드묾 |
| evidence_type 구분 | CFA V(A) 준수 | 사실/추정/의견 구분은 CFA 표준이나 자동화 시스템은 드묾 |

### 부족한 것 — 5개 차원

```
현재                              Top-Tier
  │                                  │
  │  ① 검증 신뢰성 ──────────────── 정량적 캘리브레이션
  │  ② 데이터 깊이 ──────────────── 대안 데이터 + 실시간
  │  ③ 방법론 엄밀성 ─────────────── 통계적 검증 + 인과 추론
  │  ④ 운영 성숙도 ──────────────── 연속 모니터링 + 자동 알림
  │  ⑤ 거버넌스 ─────────────────── 감사 추적 + 다중 검토자
```

## 차원 1: 검증 신뢰성 (Calibration)

### 현재 문제

엔진이 🟢을 내렸을 때, 그 🟢이 실제로 얼마나 정확한지 모른다.
"우리 엔진의 🟢 판정 중 몇 %가 실제로 맞았는가?" — 이 질문에 답할 수 없다.

### Top-Tier 기준

| 기준 | 설명 | 출처 |
|---|---|---|
| Brier Score | 확률 예측의 정확도. 0에 가까울수록 좋음 | 기상예보, Superforecasting |
| 캘리브레이션 곡선 | "70% 확신했을 때 실제 70% 맞았는가" | Philip Tetlock 연구 |
| 판정 정밀도/재현율 | 🔴을 내려야 할 때 🔴을 냈는가 (Precision/Recall) | ML 표준 |

### 필요 작업

```
Level 1 (3~6개월): 사후 추적 데이터 축적
  - BLOCK 5의 verification_claim_outcomes 테이블 가동
  - 90일 후 outcome_correct 채우기 시작
  - 최소 50건 축적 → 첫 캘리브레이션 리포트

Level 2 (6~12개월): 정량적 캘리브레이션
  - 층별 정확도 산출: "① Fact의 🟢은 95% 정확, ③ Logic의 🟢은 72% 정확"
  - 규칙별 정밀도: "lr_017이 트리거됐을 때 실제 문제 비율 = 68%"
  - 캘리브레이션 곡선 생성 → 과신/과소신 영역 식별

Level 3 (12개월+): 동적 신뢰도
  - 각 판정에 신뢰구간 부착: "🟢 (85% 신뢰도, 과거 유사 검증 기반)"
  - Brier Score 기반 엔진 성능 대시보드
  - 캘리브레이션이 나빠지면 자동 경고
```

### 핵심 지표

```
목표: Brier Score < 0.15 (Superforecaster 수준)
현재: 측정 불가 (사후 추적 데이터 없음)
```

## 차원 2: 데이터 깊이 (Data Depth)

### 현재 문제

MCP 소스가 공개 데이터에 한정. Bloomberg/Refinitiv 접근 불가.
"컨센서스 전망"이라는 핵심 데이터를 구조화된 형태로 가져올 수 없다.

### Top-Tier 기준

| 소스 유형 | 글로벌 표준 | 현재 엔진 |
|---|---|---|
| 컨센서스 전망 (EPS/GDP) | Bloomberg, FactSet, Visible Alpha | ❌ Tavily 스크래핑으로 대체 |
| 실시간 시장 데이터 | Bloomberg, Reuters | ⚠️ Yahoo Finance (15분 지연) |
| 대안 데이터 (Alternative) | 위성, 웹트래픽, 특허, 채용공고 | ❌ 없음 |
| 신용 데이터 | S&P Capital IQ, Moody's | ❌ 없음 |
| ESG 데이터 | MSCI ESG, Sustainalytics | ❌ 없음 |
| 수급 데이터 | 공매도 잔고, 기관 보유, 옵션 OI | ⚠️ 부분적 (DART 대량보유) |

### 필요 작업

```
Level 1 (즉시): 기존 MCP 최대 활용
  - FRED 시리즈 확장 (현재 13개 → 30개+)
  - SEC-EDGAR 13F (기관 보유) 분석 규칙 추가
  - DART 대량보유 보고서 연동

Level 2 (3~6개월): 대안 데이터 소스 추가
  - OpenBB Platform MCP 구축 → 컨센서스 전망 구조화
  - Google Trends MCP → 관심도 추이
  - 특허 데이터 (Google Patents / WIPO) → 기술 경쟁력 검증

Level 3 (6~12개월): 프리미엄 데이터
  - Trading Economics API → 매크로 컨센서스
  - 위성 데이터 API (Planet, Orbital Insight) → 물류/재고 검증
  - 웹 트래픽 (SimilarWeb API) → SaaS/플랫폼 기업 검증
```

## 차원 3: 방법론 엄밀성 (Methodological Rigor)

### 현재 문제

모든 판정이 LLM의 정성적 판단에 의존. 통계적 근거가 없다.
"MVRV > 3.0이면 과열" — 이 임계값의 통계적 근거는? 과거 데이터에서 검증된 것인가?

### Top-Tier 기준

| 방법론 | 설명 | 적용 영역 |
|---|---|---|
| 베이지안 업데이트 | 새 증거가 들어올 때마다 확률을 갱신 | 모든 층의 판정 |
| 인과 추론 (Causal Inference) | 상관관계 ≠ 인과관계를 정량적으로 구분 | ③ Logic KC |
| 몬테카를로 시뮬레이션 | 핵심 가정의 불확실성을 확률 분포로 표현 | DCF 밸류에이션 검증 |
| 백테스팅 | 규칙의 과거 정확도를 데이터로 검증 | 규칙 엔진 전체 |
| 앙상블 검증 | 복수 모델이 독립적으로 검증 → 불일치 탐지 | 전체 파이프라인 |

### 필요 작업

```
Level 1 (3~6개월): 규칙 백테스팅 프레임워크
  - 과거 리포트 + 실제 결과 데이터로 각 규칙의 정밀도/재현율 측정
  - 예: lr_002 (PER > 2.5배 AND 리스크 미언급) → 과거 50건 중 실제 하락 비율?
  - 임계값 최적화: MVRV 3.0이 최적인가, 2.5 또는 3.5가 더 나은가?

Level 2 (6~12개월): 확률적 판정
  - 현재: "🟢 / 🟡 / 🔴" (범주형)
  - 목표: "🟢 (P=0.82)" (확률 부착)
  - 베이지안 프레임워크: Prior(규칙 기반) × Likelihood(MCP 증거) = Posterior(최종 판정)

Level 3 (12개월+): 인과 추론 통합
  - KC 추출에 DAG(Directed Acyclic Graph) 활용
  - 인과 체인의 각 링크에 인과 강도 추정
  - 반사실 분석: "이 전제가 거짓이었다면 결론이 어떻게 바뀌었을까"를 정량화
```

## 차원 4: 운영 성숙도 (Operational Maturity)

### 현재 문제

검증이 1회성 수동 트리거. 검증 후 환경이 바뀌어도 알림이 없다.
KC가 트리거됐는지 아무도 모니터링하지 않는다.

### Top-Tier 기준

| 기능 | 글로벌 표준 | 현재 엔진 |
|---|---|---|
| 연속 모니터링 | Bloomberg Alert, Refinitiv Watch | ❌ 없음 |
| KC 자동 감시 | 헤지펀드 리스크 시스템 | ❌ KC는 검증 시점에만 확인 |
| 배치 검증 | 매일 전 포트폴리오 리포트 스캔 | ❌ 수동 1건씩 |
| 대시보드 | Grafana/Tableau 실시간 | ❌ 없음 |
| 알림 시스템 | Slack/이메일 자동 알림 | ❌ 없음 |

### 필요 작업

```
Level 1 (3~6개월): KC 워치독 (Watchdog)
  - 검증 완료 후 KC 전제를 watchlist에 등록
  - 주기적으로 MCP 조회 → KC 상태 변화 감지
  - 변화 발생 시 "재검증 필요" 알림

Level 2 (6~12개월): 연속 모니터링 파이프라인
  - invalidation_triggers의 이벤트 날짜 도래 시 자동 재검증
  - FOMC 회의 → 매크로 관련 모든 검증 결과 자동 재평가
  - Notion 아카이브에 상태 자동 업데이트

Level 3 (12개월+): 포트폴리오 수준 검증
  - 개별 문서가 아닌 포트폴리오 전체의 노출 리스크 평가
  - 섹터 간 상관관계를 고려한 복합 리스크
  - "이 3개 리포트가 모두 같은 매크로 전제에 의존하고 있다" 탐지
```

## 차원 5: 거버넌스 (Governance)

### 현재 문제

단일 LLM이 검증하고 단일 LLM이 자기 점검. 실질적 독립 검증이 없다.
감사 추적(Audit Trail)이 JSON 파일 수준.

### Top-Tier 기준

| 기준 | 설명 | 출처 |
|---|---|---|
| 4-Eyes Principle | 최소 2명(또는 2개 시스템)이 독립 검토 | 금융 규제 표준 |
| Audit Trail | 모든 판정의 근거와 과정을 추적 가능 | SOX, GIPS |
| Model Risk Management | 모델 자체의 리스크를 평가하고 관리 | SR 11-7 (Fed) |
| Adversarial Testing | 의도적으로 오류를 삽입하여 탐지 능력 측정 | Red Team 표준 |

### 필요 작업

```
Level 1 (3~6개월): 앙상블 검증
  - 동일 문서를 2개 모델(예: Claude + GPT)로 독립 검증
  - 판정 불일치 시 → "검증자 간 불일치" 플래그
  - 불일치율이 높은 층 → 해당 층의 방법론 재검토

Level 2 (6~12개월): Red Team 테스트
  - 의도적으로 오류를 삽입한 "함정 문서" 제작
  - 엔진이 오류를 탐지하는 비율 측정 (Detection Rate)
  - 탐지 실패 사례 → 규칙/프롬프트 보강
  - 목표: Detection Rate > 90%

Level 3 (12개월+): 규제 수준 감사 추적
  - 모든 MCP 호출의 요청/응답 로깅
  - 판정 변경 이력 (누가, 언제, 왜 바꿨는지)
  - 외부 감사인이 검증 과정을 재현할 수 있는 수준의 문서화
```

## 차원 6: 제품화 (Productization)

### 현재 상태

개인 투자자 1명을 위한 MCP 도구. 확장성·재사용성이 없다.

### Top-Tier로 가려면

```
Level 1: API화
  - 검증 요청 → JSON 결과 반환 REST API
  - 외부 시스템(노션 자동화, 슬랙 봇)에서 호출 가능

Level 2: 다중 사용자
  - 사용자별 커스텀 규칙/체크리스트
  - 팀 내 검증 결과 공유 + 코멘트
  - 권한 관리 (검증자 / 리뷰어 / 관리자)

Level 3: 마켓플레이스
  - 섹터별 규칙 팩 (반도체 전문가가 만든 규칙을 바이오 분석가가 구독)
  - 커뮤니티 기반 체크리스트 개선
  - 검증 결과의 익명화된 벤치마크 ("이 리포트는 상위 20% 품질")
```

## 글로벌 Top-Tier의 정의

이 엔진이 "글로벌 Top-Tier"라고 말할 수 있으려면:

| 기준 | 목표값 | 측정 방법 |
|---|---|---|
| **판정 정확도** | 🟢 판정의 90%+ 실제 정확 | outcome_correct 사후 추적 |
| **Brier Score** | < 0.15 | 확률 판정 도입 후 측정 |
| **탐지율** | Red Team 오류의 90%+ 탐지 | 함정 문서 테스트 |
| **캘리브레이션** | 70% 확신 판정의 실제 정확도 65~75% | 캘리브레이션 곡선 |
| **규칙 커버리지** | 주요 15개 섹터 전체 Omission 구축 | 체크리스트 갭 0 |
| **응답 시간** | KC 트리거 발생 → 24시간 내 알림 | KC 워치독 |
| **감사 추적** | 모든 판정의 근거를 6개월 후에도 재현 가능 | 외부 감사 테스트 |

## 솔직한 평가

현재 엔진은 **"구조는 Top-Tier급이지만, 검증된 실적이 없다"** 는 상태입니다.

프레임워크(6층 + KC + BBJ + evidence_type)는 학술적·실무적으로 견고합니다.
그러나 "이 프레임워크가 실제로 얼마나 정확한가"를 증명하는 데이터가 0건입니다.

**가장 시급한 것은 화려한 기능 추가가 아니라, 50건의 검증 + 사후 추적으로 "우리 엔진의 🟢이 진짜 맞았다"를 증명하는 것입니다.**
