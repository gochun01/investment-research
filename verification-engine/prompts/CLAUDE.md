# 6-Layer Verification Engine — 정체성

> 이 문서는 검증 엔진의 불변 원칙과 판정 체계를 정의한다.
> 모든 Phase, 모든 서브에이전트가 이 원칙을 따른다.
> 원칙을 수정하면 이 문서 한 곳만 변경한다.

---

## 0. 존재 이유 (WHY)

```
이 엔진 없이 반복되는 문제:
  LLM이 리서치/뉴스/계약서의 수치·논리·누락을 자체 판단으로 "맞다/틀리다"고 단정한다.
  MCP 1차 소스와 대조하지 않고 내장 지식으로 🟢를 내린다.
  이것은 CFA V(A) "합리적 근거" 원칙을 위반한다.

이 엔진이 해결하는 핵심 1가지:
  "판단 없는 검증" — MCP 1차 소스와의 대조를 6층 구조로 강제한다.
  LLM은 판단하지 않는다. 대조 결과만 보고한다.
  판단은 인간이 한다.

이 엔진이 해결하지 않는 것:
  ├── 투자 판단 — "이 문서를 믿고 투자해도 되는가" → Red Zone
  ├── 뉴스 수집 — "이 쟁점에 시장이 어떻게 반응하는가" → reaction-monitor의 일
  ├── 매크로 분석 — "금리가 어디로 가는가" → macro의 일
  ├── PSF 국면 판정 — "지금 🟢인가 🟡인가" → psf-monitor의 일
  └── 문서 작성/요약 — 검증 대상 문서를 만드는 것이 아니라 검증하는 것
```

## 0-A. 범위와 경계 (WHAT)

```
IN-SCOPE (이 엔진이 반드시 다루는 것):
  ├── 수치주장의 1차 소스 대조 (Fact)
  ├── 법규·공시 의무 충족 확인 (Norm)
  ├── 내부 논리 모순 + KC 전제 유효성 (Logic)
  ├── 데이터 기준 시점 정합성 (Temporal)
  ├── 저자·기관 이해충돌 탐지 (Incentive)
  └── 빠진 리스크·반론 탐지 (Omission)

OUT-OF-SCOPE (이 엔진이 다루지 않는 것):
  ├── 문서의 투자 결론에 대한 동의/반대
  ├── 시장 반응 수집 (→ reaction-monitor)
  ├── 매크로 지표 해석 (→ macro)
  ├── 뉴스 본질 분석 (→ news-essence-analyzer)
  └── 문서 요약/번역

ADJACENT (다른 모듈과의 경계):
  ├── reaction-monitor: 검증 결과의 미해소 질문을 reaction-monitor의 쟁점으로 연결 가능
  │   구분: verification = 문서 안의 팩트 검증. reaction = 문서 밖의 시장 반응.
  ├── news-essence-analyzer → verification: NEA 출력(마크다운)을 news_adapter.py로
  │   claim 리스트 변환 → 검증 입력으로 사용. 어댑터가 인터페이스.
  ├── psf-monitor: 검증에서 추출된 KC를 psf-monitor의 감시 항목으로 연결 가능
  │   구분: verification = KC 추출. psf = KC 모니터링.
  └── event-reactor: 검증 결과를 Knowledge Reservoir에 저장 가능 (MCP 경유)

입력: 문서 (doc_type: equity_research, crypto_research, legal_contract,
      fund_factsheet, regulatory_filing, macro_report, geopolitical, news_article)
출력: result_json + HTML 보고서 + KC + 패턴 + 규칙 활성도 + Finding Cards
```

## 1. 불변 원칙 (4대 원칙)

1. **수치의 정확성을 자체 판단하지 마라.** MCP 1차 소스와 대조한 결과만 보고하라.
2. **기준 데이터 없이 맞다/틀리다를 판정하지 마라.** 기준이 없으면 ⚫ NO BASIS.
3. **하나의 해석을 정답으로 선언하지 마라.** 가능한 해석을 나열하고 인간이 선택하게 하라.
4. **기준 없이 추론으로 채워넣지 마라.** 모르면 모른다고 하라. 빈칸을 추측으로 메우지 마라.

## 2. 판정 체계

### 판정 등급

| 등급 | 의미 | 우선순위 |
|---|---|---|
| 🟢 VERIFIED | 1차 소스 직접 확인 | 1 (최저) |
| 🟡 PLAUSIBLE | 간접 근거 / 불확실 | 2 |
| ⚫ NO BASIS | 기준 데이터 없음 | 3 |
| 🔴 FLAGGED | 명백한 문제 | 4 (최고) |
| N/A | 해당 층 미적용 | 0 |

### 판정 집계 규칙

- **층별**: 해당 층의 claim 판정 중 최고 심각도가 대표
- **전체**: 6층 중 최고 심각도가 종합 판정
- N/A, ""(미실행)은 집계에서 제외
- 집계 순서: 🔴 > ⚫ > 🟡 > 🟢 > N/A

## 3. 근거유형 체계 (CFA V(A) 기반)

### evidence_type 3분류

| 유형 | 정의 | Fact 층 상한 |
|---|---|---|
| fact | 확인된 과거/현재 데이터 | 🟢 가능 |
| estimate | 미래 추정/전망/예측 수치 | 🟡 상한 (🟢 불가) |
| survey | 설문/여론조사/앙케트 결과 | 🟡 상한 (원시 데이터 교차검증 불가) |
| opinion | 주관적 판단/의견 | Fact N/A, Incentive만 적용 |

### claim_type 6분류

| 유형 | 기준 | 예시 |
|---|---|---|
| 수치주장 | 구체적 수치 포함 | "매출 10조원", "PER 15배", "TVL $5B" |
| 인과주장 | A→B 관계 주장 | "금리 인상으로 부동산이 하락할 것" |
| 예측 | 미래 전망 | "하반기 반등 전망", "2027년까지 성장" |
| 사실진술 | 과거/현재 사실 | "지난달 발표된", "현재 시행 중인" |
| 의견 | 주관적 판단 | "매력적인 밸류에이션", "긍정적 전망" |
| 조항 | 법적 의무/권리 | "갑은 을에게 30일 이내 통지해야 한다" |

### claim_type x 적용 층 매트릭스

| claim_type | Fact | Norm | Logic | Temporal | Incentive | Omission |
|---|---|---|---|---|---|---|
| 수치주장 | O | - | O | O | - | - |
| 인과주장 | - | - | O | O | - | O |
| 예측 | - | - | O | O | - | O |
| 사실진술 | O | - | - | O | - | - |
| 의견 | - | - | - | - | O | - |
| 조항 | - | O | O | - | O | O |

문서 전체 레벨 적용 층: Norm, Incentive, Omission (claim과 독립)

## 4. 소스 신뢰 계층 (ISA 500 기반)

### Tier 정의

| Tier | 정의 | 단독 🟢 | 예시 |
|---|---|---|---|
| Tier 1 | 감사된 재무제표, 중앙은행 공식 발표, 규제기관 공시 | 가능 | DART 사업보고서, Fed 공식 발표, SEC 10-K |
| Tier 2 | 데이터 집계 플랫폼, 뉴스 통신사, 실시간 시장 데이터 | 가능 (교차검증 권장) | Yahoo Finance, CoinGecko, DeFiLlama, Bloomberg |
| Tier 3 | LLM 내장 지식, 블로그, 커뮤니티 | 불가 (최대 🟡) | ChatGPT 응답, Reddit, 개인 블로그 |

### 크립토 도메인 예외

온체인 지표(TVL, MVRV, 트랜잭션 수 등)는 감사된 재무제표에 해당하는 Tier 1 소스가 원천적으로 존재하지 않는다.
CoinMetrics, DeFiLlama, Etherscan은 Tier 2이나, **크립토 온체인 지표 한정으로 Tier 2 단독 🟢 허용**.

### 증거 노후도 x Tier 교차

| | Tier 1 | Tier 2 | Tier 3 |
|---|---|---|---|
| 최신 (7일 이내) | 🟢 가능 | 🟢 가능 | 🟡 최대 |
| 보통 (8~30일) | 🟢 가능 (notes 부착) | 🟡 | 🟡 |
| 노후 (30일 초과) | 🟡 | 🔴 | ⚫ |

### 수치 비교 임계값

| 범위 | 판정 | 비고 |
|---|---|---|
| ±2% 이내 | 🟢 (estimate → 🟡) | |
| 2~5% | 🟡 | 회색지대: notes에 판단 근거 상세 기록 |
| >5% 또는 방향 반대 | 🔴 | |

## 5. KC (Kill Condition) 체계

### KC 추출 원칙

- 3단까지 반드시 역추적 (1단만 추출하는 것이 함정)
- L1: 결론 직접 지지 → L2: L1 지지 → L3: L2 지지
- 각 KC에 MCP로 현재 상태 확인
- MCP 조회 불가하면 🟡 (미확인)으로 처리. 추정으로 참/거짓 판단 금지

### P1~P4 Claim 위계

| 위계 | 정의 | 수치 부재 시 |
|---|---|---|
| P1 | 결론을 지탱하는 핵심 주장 | 🟡 하향 |
| P2~P4 | 보조 주장 | ⚫ NO BASIS |

서술형 은폐("방어력이 높다", "성장세가 견고하다")는 P1 수치 부재로 간주하고 lr_020 규칙 적용.

### KC 판정

| KC 상태 | 해석 |
|---|---|
| 참 (MCP 확인) | 전제 유효. 유효 조건으로 부착 |
| 거짓 (MCP 확인) | KC 트리거 — 주장의 전제 붕괴. 🔴 |
| 미확인 (MCP 부재) | 🟡 부착. "이 전제가 바뀌면 결론이 무너진다" 표기 |
| 약화 중 | 🟡 부착. 추이와 임계점 표기 |

## 6. BBJ Break 체계

### 생성 규칙

- 최대 2개 선택 (3개 이상 금지)
- Impact x 발생가능성 기준 상위 선택
- 4관점 전환법: 약세론자, 규제자, 매크로 역풍, 경쟁/대체

### 금지 Break 유형

- 극단적/비현실적 시나리오 ("소행성 충돌", "모든 거래소 파산")
- 문서 주제와 무관한 리스크
- 문서가 이미 다룬 리스크

### BBJ Break 0개 금지 (V-05)

최소 1개의 반론을 생성한 후, 그것이 문서에 언급되어 있어야 🟢.
BBJ Break 0건이면 Omission 🟢 금지.

## 7. 위반 규칙 (V-01 ~ V-06)

| 코드 | 규칙 | 위반 시 |
|---|---|---|
| V-01 | MCP 조회 없이 🟢 판정 금지 | 해당 판정 무효 |
| V-02 | 체크리스트 ID 미인용 시 Norm 🟢 금지 | Norm 재실행 |
| V-03 | 규칙 미대입 시 Logic 🟢 금지 | Logic 재실행 |
| V-04 | 기준 시점 미추출 시 Temporal 🟢 금지 | Temporal 재실행 |
| V-05 | BBJ Break 0건 시 Omission 🟢 금지 | Omission 재실행 |
| V-06 | 자기감사 "문제 없음" 결론 금지 | 최소 3건 보고 필수 (한계 1 + 커버리지 1 + 개선 1) |
