## 불변 원칙

1. **판정을 재평가하지 않는다.** 이미 내린 🟢🟡🔴⚫는 그대로 유지한다.
2. **자동 수정하지 않는다.** "고치겠습니다"가 아니라 "이런 문제가 있었습니다"까지만.
3. **조치 결정은 사용자 몫이다.** 시스템은 문제를 보고하고, 사용자가 재검증/수정을 지시한다.

### ★ V-06: "문제 없음" 종료 금지

```
자기 점검에서 "모든 항목 문제 없음"으로 끝나는 것을 금지한다.

이유: 검증 엔진 자체가 완벽할 수 없다. 어떤 검증이든 반드시 한계가 존재한다.
"문제 없음"은 "문제를 발견하지 못했다"와 다르다.

필수 최소 보고:
  ① 검증 한계 — 최소 1건 (조회하지 못한 소스, 미구축 체크리스트, Tier 3 의존 등)
  ② 도메인 커버리지 갭 — 해당 문서의 doc_type/sector에 대한 규칙·체크리스트 부족 여부
  ③ 수정 권유 — 최소 1건 (체크리스트 추가 권장 또는 규칙 추가 권장)

모든 층이 🟢이더라도 위 3가지는 반드시 보고하라.
"이번 검증은 완벽했다"는 존재하지 않는다.
```

## 점검 항목 9가지 (기존 7 + 신규 2)

### 1. MCP 조회 누락

```
확인:
- evidence 필드가 비어있는 🟢 판정이 있는가? → V-01 위반 경고
- 조회해야 했지만 MCP 도구가 없어서 ⚫ 처리한 수치가 있는가?
- 같은 MCP 소스를 중복 조회한 곳이 있는가?
```

### 2. 체크리스트 부족

```
확인:
- ② Norm에서 해당 도메인 체크리스트가 없어서 ⚫ 처리한 항목이 있는가?
- ⑥ Omission에서 해당 섹터 체크리스트가 없어서 유사 섹터를 참조한 경우?
- 이번 검증에서 "이 항목이 체크리스트에 있었으면 좋겠다"는 것?
```

### 3. 규칙 엔진 부족

```
확인:
- ③ Logic에서 적용할 규칙이 없어서 "적용 불가"로 처리한 경우?
- 이번 문서에서 발견한 논리적 패턴 중 규칙으로 추가되면 좋겠는 것?
```

### 4. 검증 한계 (Limitation Disclosure)

```
솔직하게 서술:
- 가장 불확실한 판정은 어느 claim의 어느 층인가? (판정 변경 아님, 불확실성 명시)
- 1차 소스 대신 뉴스/2차 소스에 의존한 부분은?
- 문서의 도메인이 체크리스트/규칙 커버리지 밖에 있는 부분은?
```

### 5. ★ 신뢰도 캘리브레이션 (Self-Confidence Audit)

```
각 층의 판정에 대해 "이 판정이 틀릴 확률"을 자문하라:

① Fact: MCP 조회를 했으므로 보통 신뢰도가 높다.
   하지만: 단위 불일치를 놓쳤을 가능성은? 교차검증을 했는가?

③ Logic: KC 추출이 가장 주관적이다.
   자문: "내가 추출하지 못한 KC가 있는가?" "3단까지 갔는가, 1단에서 멈췄는가?"
   자문: "논리 오류 패턴을 탐지했는가, 문서의 논조에 끌려갔는가?"

⑥ Omission: BBJ Break의 품질이 핵심이다.
   자문: "4관점 전환을 모두 시도했는가?" "문서의 결론에 동조하여 약한 Break만 만들지 않았는가?"
   자문: "동심원 검사에서 매크로 전제를 정말 MCP로 확인했는가?"

신뢰도가 낮은 판정이 있으면 솔직하게 표기하라:
"③ Logic/c005: KC 추출이 2단에서 멈춤. 3단 전제(지속 가능성)는 MCP 소스 부재로 미확인."
```

### 6. ★ 메타인지 체크 (LLM 편향 자가진단)

```
아래 질문에 솔직하게 답하라:

□ "문서의 결론에 동조하지 않았는가?"
  → 문서가 "매수"이면 나도 모르게 긍정적 판정 편향이 생길 수 있다.
  → 🟢 판정 비율이 80% 이상이면 경고 표시.

□ "앞선 층의 판정에 끌려가지 않았는가?"
  → ① Fact가 🟢이면 ③ Logic도 관대해지는 경향.
  → 각 층은 독립 판정. 앞 층 결과를 참조하되 끌려가서는 안 된다.

□ "어려운 판단을 🟡로 회피하지 않았는가?"
  → 🟡는 "불확실"이지 "판단 회피"가 아니다.
  → 🟡 판정 비율이 50% 이상이면: 판단을 회피하고 있을 가능성. 경고 표시.

□ "⚫를 남발하지 않았는가?"
  → ⚫는 "1차 소스 부재"이지 "조사하기 귀찮다"가 아니다.
  → ⚫ 처리한 항목에 대해: 정말 MCP 도구가 없었는가? WebSearch라도 시도했는가?
```

### 7. 다음 검증을 위한 권장 사항

```
같은 유형 문서를 다음에 검증할 때 먼저 확인해야 할 것:
- 이번에 부족했던 MCP 소스
- 이번에 부족했던 체크리스트 항목
- 이번에 부족했던 규칙
```

### 8. ★ 도메인 커버리지 갭 보고 [필수]

```
이번 검증의 doc_type과 sector에 대해 아래를 보고하라:

① 전용 규칙 커버리지:
   - 이 doc_type에 전용 규칙이 몇 개 있는가? (common 제외)
   - 전용 규칙이 0개이면: "⚠ 전용 규칙 미구축. common 규칙만 적용됨."

② Omission 체크리스트 커버리지:
   - 이 sector에 Omission 항목이 몇 개 있는가?
   - 0개이면: "⚠ 섹터 Omission 체크리스트 미구축. 빠진 리스크 탐지가 BBJ Break에만 의존."

③ 금융 특화 규칙 부재 고백:
   아래 중 이번 검증에 해당하는 것이 있으면 솔직하게 보고:

   [밸류에이션 검증 한계]
   - DCF 터미널 성장률 적정성 검증 규칙 없음
   - PER 밴드 분석, EV/EBITDA 비교 규칙 없음
   - Sum-of-Parts 밸류에이션 정합성 규칙 없음

   [채권/크레딧 한계]
   - 신용등급 변동 리스크, 회사채 스프레드 검증 규칙 없음
   - 코베넌트(Covenant) 분석 규칙 없음

   [한국 시장 특화 한계]
   - K-IFRS vs US-GAAP 차이에 의한 수치 비교 주의 규칙 없음
   - 한국 주요 섹터(2차전지, 방산, 조선, 자동차, 은행/보험, 리츠) Omission 미구축
   - 원/달러 환율 영향 분석 규칙 없음

   [파생상품/구조화상품]
   - 완전 미지원. 해당 문서가 들어오면 ⚫ 대량 발생 예상

   [펀드/ETF 분석]
   - fund_factsheet doc_type은 규칙·체크리스트 미구축 (대부분 ⚫ 처리)

해당 없는 항목은 생략. 해당하는 항목만 보고하되, 최소 1건은 반드시 보고.
```

### 9. ★ 수정 권유 [필수]

```
이번 검증에서 발견한 구체적 개선 제안을 최소 1건 제시하라.
사용자가 "체크리스트 추가해줘" 또는 "규칙 추가해줘"로 즉시 반영할 수 있는 형태로.

형식:
  [체크리스트 추가 권장]
  - {checklist_type}/{sector}: "{항목}" — {이유} — {심각도}

  [규칙 추가 권장]
  - {doc_type}/lr_NNN: "{조건}" → "{플래그}" — {이유}

  [MCP 소스 추가 권장]
  - {소스명}: {커버하는 데이터} — {이유}

★ "이번 검증에서 개선할 점이 없다"는 허용되지 않는다.
  최소한 "이 섹터에 OO 항목이 있으면 다음 검증에서 유용하겠다" 수준이라도 제시하라.
```

## 피드백 루프 규칙

```
자기 점검 + 검증 결과 요약 제시
      │
      ▼
  사용자 판단
      │
      ├── "괜찮아" / "진행해" → Phase 5 (HTML 보고서) 진행
      │
      ├── "재검증해줘" → 해당 층 재실행 → finalize 재호출 → 결과 재제시
      │
      ├── "체크리스트 추가해줘" → verify_add_checklist_item() → 해당 층 재실행 → 결과 재제시
      │
      ├── "규칙 추가해줘" → verify_add_rule() → 해당 층 재실행 → 결과 재제시
      │
      └── "판정 수정해줘" → verify_set_verdict() → finalize 재호출 → 결과 재제시
```

★ **수정 시 반드시 verify_finalize()를 재호출하여 result_json을 갱신한다.**
★ 갱신된 result_json은 output/history/에 자동 재저장된다.
★ **사용자가 "괜찮아"/"진행해"라고 명시적으로 승인하기 전까지 Phase 5로 넘어가지 마라.**
