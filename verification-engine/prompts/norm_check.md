# ② Norm Ground 검증 프롬프트

## 원칙

> 4대 불변 원칙, 판정 체계의 전체 정의는 **CLAUDE.md** 참조.
> 아래는 ② Norm Ground 전용 규칙만 기술한다.

---

아래 체크리스트의 각 항목이 문서에 존재하는지 **키워드 스캔**하여 검증하라.
이것은 **문서 전체 레벨** 판정이다.

## ② Norm 전용 원칙

- **체크리스트 미참조 판정 금지** (V-02). "일반적으로 리포트에는 이런 항목이 있다"는 대조가 아니다.
- 반드시 아래 체크리스트의 **구체적 item_id를 인용**해야 한다.
- 체크리스트가 없는 도메인이면 ⚫ NO BASIS.

## 체크리스트

{checklist}

> 각 항목은 `{id, description, severity, scan_keywords, regulation}` 구조의 JSON 배열이다.
> 상세 스키마는 `schemas/io-contracts.md` 참조.

---

## 스캔 방법

1. 각 항목의 scan_keywords를 문서에서 검색
2. 키워드가 발견되면: 해당 문장의 위치를 기록 → **matched**
3. 키워드가 없으면: **missing**
4. 키워드가 발견되었으나 형식적이거나 맥락 불일치 → **partial** (아래 "형식 vs 실질 구분" 참조)

---

## ★ 형식 vs 실질 구분

키워드가 발견되었더라도 **실질적 충족인지 형식적 존재인지** 구분하라.

| 상태 | 예시 | 분류 | 처리 |
|---|---|---|---|
| 실질 충족 | "당사는 주간사로 참여하였으며, 자기매매 부서에서 000주 보유" | **matched** | checklist_matched에 포함 |
| 형식적 존재 | "이해관계가 있을 수 있습니다" (구체적 내용 없음) | **partial** | checklist_matched에 포함하되, notes에 "⚠ [item_id] 형식적 기재, 실질 내용 부재" 기록. **에스컬레이션에서 medium 누락 1건으로 취급** |
| 키워드 있으나 다른 맥락 | "리스크"라는 단어가 있으나 투자 리스크 경고가 아닌 사업 설명 | **missing** | checklist_missed에 포함. notes에 "키워드 존재하나 맥락 불일치" 기록 |

### 맥락 판단 기준

키워드가 문서에 있을 때, 아래 질문으로 실질 충족 여부를 판단하라:
- 이 문장이 **해당 항목의 description이 요구하는 정보를 실제로 제공**하고 있는가?
- **구체적 수치·주체·조건**이 명시되어 있는가, 아니면 일반론인가?
- 이 문장을 읽은 투자자/계약 당사자가 **판단에 필요한 정보를 얻을 수 있는가?**

---

## ★ 복수 항목 누락 시 에스컬레이션

| 누락 상황 | 판정 |
|---|---|
| **critical 1개+ 누락** | **🔴** |
| high 1개+ 누락 | 🔴 |
| medium 1~2개 누락 | 🟡 |
| medium 3개+ 누락 | 🟡 → 🔴 에스컬레이션 |
| high + medium 동시 누락 | 🔴 (복합 누락 경고) |
| low만 누락 | 🟢 (notes에 기록) |
| 형식적 존재(partial) | 해당 severity의 누락 0.5건으로 산정 (medium partial 2건 = medium 누락 1건) |

---

## 판정 기준

| 판정 | 조건 |
|---|---|
| 🟢 | 필수 항목(critical/high) 전부 실질 충족 |
| 🟡 | 부분 충족, 형식적 존재, 또는 medium 소수 누락 |
| 🔴 | 필수 항목(critical/high) 누락, 또는 medium 대량 누락 |
| ⚫ | 해당 도메인 체크리스트 미구축 |

### regulation 필드 활용

high/critical 항목 누락 시, notes에 해당 항목의 **regulation 필드를 인용**하라.

```
예: "🔴 nr_lc_001(관할 법원 조항) 누락. 관련 법규: 민법/중재법"
예: "🔴 nr_eq_001(투자 리스크 경고) 누락. 관련 법규: 자본시장법 §178"
```

→ 어떤 법규를 위반했는지 사용자가 즉시 파악할 수 있다.

## ★ Norm과 다른 층의 역할 경계

```
Norm  = "이 항목이 문서에 형식적으로 존재하는가?" (형식 검사)
Omission = "이 리스크가 문서에서 실질적으로 다뤄졌는가?" (내용 검사)

예: 리스크 경고 조항이 있지만 형식적 → Norm 🟡 + Omission에서 별도 판정
→ Norm은 형식적 존재 여부만 판정한다. 내용의 실질성은 ⑥ Omission의 영역이다.
```

### ★ Norm 🔴 시 파급 경고

Norm 🔴(필수 항목 누락)은 **문서의 전문성·신뢰도 의심 시그널**이다.

```
Norm 🔴 판정 시 notes에 반드시 부착:
  "⚠ 문서 형식 요건 미달. 전문 리서치/법률 문서로서의 기본 요건을 충족하지 않음.
   다른 층의 판정 해석 시 문서 자체의 신뢰도를 고려하라."
```

→ Norm 🔴 자체가 다른 층의 판정을 바꾸지는 않지만(독립 층 원칙),
  최종 보고서에서 사용자가 판단할 때 **문서 품질 경고**로 참조된다.

---

## 출력

`verify_set_document_verdict()`를 호출하라:

```
verify_set_document_verdict("norm",
    verdict="🟡",
    notes="nr_eq_001 실질 충족. nr_eq_002 형식적 기재(구체적 보유현황 미기재, partial). nr_eq_004 미기재. 관련 법규: 금감원 모범규준.",
    checklist_matched=["nr_eq_001", "nr_eq_002", "nr_eq_003", "nr_eq_005"],
    checklist_missed=["nr_eq_004"]
)
```

★ 형식적 존재(partial) 항목은 `checklist_matched`에 포함하되, **notes에 "partial" 표기 필수**.
