# ④ Temporal Ground 검증 프롬프트

## 원칙

> 4대 불변 원칙, 판정 체계의 전체 정의는 **CLAUDE.md** 참조.
> 아래는 ④ Temporal Ground 전용 규칙만 기술한다.

## ④ Temporal 전용 원칙

- **기준 시점 미추출 🟢 금지** (V-04). claim의 데이터가 언제 기준인지 모르면 신선도를 판정할 수 없다.
- 기준 시점을 추출하지 못하면 ⚫ NO BASIS.
- 반드시 **gap 계산 + Material Event 탐지**를 모두 수행해야 한다.

---

문서 내 수치의 **기준 시점을 추출**하고, 현재까지 **유의미한 변화가 있었는지** MCP로 확인하라.

## ★ ① Fact에서 조회한 값 재사용

이 단계에서는 ① Fact Ground에서 이미 MCP로 조회한 최신값을 **재사용**한다.
같은 MCP를 중복 호출하지 마라. 아래에 주입된 Fact 조회 결과를 기준으로 비교하라.

### Fact 조회 결과 (자동 주입)

{fact_evidence}

위 데이터의 `retrieved_at`이 MCP 조회 시점이다. 이 시점과 claim의 기준 시점을 비교하라.
Fact에서 조회하지 않은 수치만 추가 MCP 호출하라.

## 실행 절차

### Step 1. 기준 시점 추출

claim에서 기준 시점을 추출한다:

| 표현 유형 | 처리 | 예시 |
|---|---|---|
| 날짜 명시 | 그대로 사용 | "2025년 3분기", "3월 12일 기준" |
| 간접 표현 | 문서 발행일 기준으로 역산 | "최근"=발행일-7일, "지난달"=발행일-30일 |
| 불명확 | ⚫ 처리 (V-04) | 기준 시점 표기 없음 |

### Step 2. 시차(gap) 계산

기준 시점과 오늘 사이의 일수를 계산.

### Step 3. Material Event 탐지

시차 사이에 **유의미한 이벤트**가 발생했는지 확인한다. 이것이 Temporal의 핵심이다.

#### 이벤트 유형 + 정량 임계값

| 유형 | 구체적 이벤트 | 임계값 (material 기준) | MCP 탐지 |
|---|---|---|---|
| **거시경제** | FOMC 금리 결정 | 금리 변경 ±25bp 이상, 또는 전망 변화 | FRED |
| | CPI/PCE 발표 | 컨센서스 대비 ±0.3%p 서프라이즈 | FRED |
| | 고용보고서/GDP | 컨센서스 대비 ±10% 서프라이즈 | FRED |
| **기업** | 실적발표 | EPS 컨센서스 대비 ±10% 서프라이즈 | DART/SEC-EDGAR |
| | 수시공시(8-K) | M&A, CEO 교체, 유상증자, 분할 | DART/SEC-EDGAR |
| | 신용등급 변경 | 1노치 이상 변동 | Tavily |
| **시장** | VIX 급등 | VIX >25 | Yahoo Finance |
| | 주요 지수 급락 | 1일 -5% 이상 | Yahoo Finance |
| **크립토** | 해킹/익스플로잇 | 피해 >$10M | CoinGecko/DeFiLlama |
| | 디페그 | 스테이블코인 ±2% 이탈 | CoinGecko |
| | 프로토콜 업그레이드 | 하드포크, 토크노믹스 변경 | Tavily |
| **정책/지정학** | 관세, 제재, 무력 충돌 | 신규 발동/철폐 | Tavily |

### Step 4. 판정

아래 **doc_type별 gap 임계값**과 Material Event 결과를 조합하여 판정하라.

## ★ doc_type별 gap 임계값

문서 유형에 따라 "신선함"의 기준이 다르다. 동일한 7일/30일 기준을 모든 doc_type에 적용하지 마라.

| doc_type | 🟢 (신선) | 🟡 (보통) | 🔴 (노후) | 근거 |
|---|---|---|---|---|
| **equity_research** | ≤7일 | 8~30일 | >30일 | 분기 실적 사이클 기준 |
| **crypto_research** | ≤3일 | 4~14일 | >14일 | 크립토 변동성 (TVL 하루 50% 변동 가능) |
| **macro_report** | ≤7일 | 8~30일 | >30일 | FOMC/CPI 발표 주기 기준 |
| **geopolitical** | ≤3일 | 4~14일 | >14일 | 지정학 이벤트 급변성 |
| **legal_contract** | ≤30일 | 31~180일 | >180일 | 법률 변경 주기 |
| **fund_factsheet** | ≤7일 | 8~30일 | >30일 | 시장 가격 기준 |

### 판정 매트릭스

| 판정 | 조건 |
|---|---|
| 🟢 | gap이 doc_type별 🟢 범위 + Material Event 없음 |
| 🟡 | gap이 doc_type별 🟡 범위 + Material Event 없음, 또는 이벤트 있으나 임계값 미달(방향성 변화 없음) |
| 🔴 | gap이 doc_type별 🔴 범위, 또는 gap 내 Material Event 발생(임계값 초과, 방향성 변화), 또는 material_change=True |
| ⚫ | 문서에 기준 시점이 명시되지 않음 (V-04) |

## ★ 복합 노후화 판정

하나의 문서에 **서로 다른 시점의 데이터**가 섞여 있을 수 있다.

예: 매출은 2025Q3(6개월 전), 주가는 어제, 금리 전망은 3개월 전

이 경우 **claim별로 개별 판정**하되, 문서 전체의 "가장 오래된 핵심 데이터"를 notes에 명시하라.

```
예시:
notes: "문서 내 시점 혼재: 매출 데이터(2025Q3, gap=180일, 🔴) /
주가(2026-03-16, gap=1일, 🟢) / 금리 전망(2025-12, gap=90일, 🟡).
핵심 결론의 근거인 매출 데이터가 가장 오래됨."
```

---

## ★ 연쇄 노후화 (depends_on 활용)

claim A가 claim B에 의존하고(depends_on), **claim A의 시점은 최신이지만 claim B의 시점이 오래됐으면**: claim A도 실질적으로 노후화된 것이다.

```
예:
c001: "매출 10조원" (2025Q3 기준) → temporal 🔴
c002: "매출 성장으로 주가 반등 전망" (depends_on: c001)
→ c001이 🔴이므로, c002도 "근거 데이터 노후화"로 🟡 이상
```

판정 시 depends_on 체인을 확인하고, 의존하는 claim의 temporal 판정이 🔴이면 **피의존 claim도 최소 🟡로 격상**하라.

### 복합 + 연쇄 동시 적용

둘 다 해당하는 경우, **더 심각한 쪽을 채택**한다.

```
예: c002가 복합 노후화로 🟡이고, 연쇄 노후화(c001 🔴)로도 🟡이면 → 🟡 유지
    c002가 복합 노후화로 🟡이고, 연쇄 노후화(c001 🔴 + material event)로 🔴이면 → 🔴
```

## 유효 기간 부착

모든 🟢 판정에 "언제까지 유효"를 부착:

| 문서 유형/범위 | 유효 기간 | 무효화 이벤트 |
|---|---|---|
| 개별종목 | 다음 분기 실적발표 전 | 실적발표, 수시공시, M&A |
| 매크로 | 1개월 | FOMC, CPI, 고용보고서 |
| 섹터 | 2개월 | 산업 정책 변경, 핵심 기업 실적 |
| 크립토/토큰 | 2주 | 해킹, 업그레이드, 규제 조치 |
| 법률 계약 | 계약 종료일 또는 1년 | 법 개정, 규제 변경 |

## ★ 검증 결과 전체의 유효 기간 관리 (NIST AI RMF 기반)

> 개별 claim의 temporal 판정과 **별도로**, 검증 결과 전체(verification result)의 유효 기간을 관리한다.
> claim이 아무리 🟢이어도, 검증 이후 환경이 바뀌면 **검증 결과 자체가 무효**가 될 수 있다.

### 자동 무효화 트리거 이벤트

아래 이벤트가 검증 완료일 이후 발생하면, **검증 결과 전체를 재검증 대상으로 격하**한다.

| 트리거 유형 | 구체적 이벤트 | 영향 범위 |
|---|---|---|
| **매크로 레짐 전환** | FOMC 금리 결정 변경, CPI 서프라이즈 (±0.3%p), 실업률 급변 | 전체 검증 |
| **기업 Material Event** | 어닝 서프라이즈 (±10%), 수시공시(8-K), M&A, CEO 교체 | 해당 기업 관련 claim |
| **시장 구조 변화** | VIX >30 (공포), 주요 지수 1일 -5% 이상, 서킷브레이커 | 전체 검증 |
| **크립토 이벤트** | 해킹/익스플로잇 (>$10M), 스테이블코인 디페그, 하드포크 | 해당 프로토콜 관련 claim |
| **규제 이벤트** | SEC 소송, 금감원 제재, 신규 법안 통과 | 해당 규제 대상 claim |
| **지정학 이벤트** | 관세 부과/철폐, 제재 발동, 무력 충돌 발생/종료 | 전체 검증 |

### 검증 결과 유효 기간 산정

```
검증 결과 전체의 유효 기간 = min(
    문서 유형별 기본 유효 기간,
    가장 가까운 예정 트리거 이벤트까지의 기간
)

예:
  문서 유형: 개별종목 → 기본 유효: 다음 분기 실적발표 전
  가장 가까운 트리거: FOMC 3/19 (2일 후)
  → 검증 유효 기간: 2일 (FOMC 결과 확인 전까지)
```

### ★ upcoming_triggers → finalize 전달

이 단계에서 식별한 트리거 이벤트는 **finalize.md의 Step 2에서 참조**한다.

전달 방법:
- notes에 upcoming_triggers를 텍스트로 기록
- valid_until 필드에 가장 가까운 트리거 날짜를 설정
- finalize가 result_json에서 각 claim의 valid_until과 notes를 읽어 전체 유효 기간을 산정

```
upcoming_triggers (notes에 기록):
  - FOMC 금리 결정: 2026-03-19 → 매크로 전제 재평가 필요
  - SK하이닉스 1Q26 실적발표: 2026-04-24 → 핵심 재무 데이터 갱신
```

verify_set_verdict("c001", "temporal",

    notes="gap 5일 (equity 🟢 범위). 기간 내 FOMC 없음, CPI 발표(3/14) 있으나 컨센서스 부합(3.1%, 서프라이즈 <0.3%p). 유의미한 변화 없음. upcoming_triggers: FOMC 3/19, SK하이닉스 실적 4/24."
)
