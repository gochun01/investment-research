## 실행 A: 규칙 대입

1. 문서에서 규칙에 필요한 수치를 추출
2. 조건에 대입
3. 조건 충족 시 플래그 발생
4. **수치 부재 처리 규칙** (기존 "적용 불가" 대체):
   - 규칙 대입에 필요한 수치를 추출할 수 없는 경우:
     → **P1 Claim**에 해당하면: **🟡 PLAUSIBLE로 하향**
     → **P2~P4 Claim**에 해당하면: **⚫ NO BASIS** 표기 후 통과
   - "결론을 지탱하는 수치가 서술형으로만 표현된 경우"는
     P1 수치 부재로 간주하고 **lr_020 규칙 적용**
   - **추정값으로 대입하지 않는다.** 빈칸을 추측으로 메우지 마라.
5. lr_000(자체 모순)은 항상 적용: 같은 문서 내 상충하는 주장 쌍 탐색

### ★ P1/P2~P4 분류 기준

claim이 P1인지 P2~P4인지 아래 기준으로 판단하라:

| 기준 | P1 (핵심) | P2~P4 (보조) |
|---|---|---|
| **depends_on** | `depends_on=[]` (다른 claim에 의존 안 함) | 다른 claim에 의존 |
| **위치** | 문서 결론부·요약·투자의견에 위치 | 본문 중간·근거 섹션에 위치 |
| **역할** | 최종 판단의 직접 근거 (목표가, 투자의견, 핵심 전망) | 핵심 주장을 뒷받침하는 보조 근거 |
| **모호하면** | — | **P2 기본값** (보수적 처리) |

### ★ 확장 규칙 처리 (context_branches / data_sources)

일부 규칙은 단순 조건-플래그 구조가 아니라 **분기 판정 구조**를 갖는다.

```
확장 규칙 식별: 규칙 JSON에 "context_branches" 키가 있으면 확장 규칙이다.

처리 절차:
1. 기본 condition을 먼저 확인 (예: "MVRV > 3.0 AND 매수 추천")
2. 기본 condition이 충족되면 → context_branches를 순서대로 평가
3. ★ 복수 branch 매칭 시: 가장 높은 severity의 branch를 적용
   (첫 매칭이 아니라, 전체 평가 후 최고 severity 선택)
4. 어떤 branch에도 해당하지 않으면 → 기본 severity로 판정

예시 (lr_007_v2):
  기본 조건: MVRV > 3.0 AND 매수 추천 → 충족됨
  branch 1: 사이클 경과 > 18개월 → CoinMetrics/CoinGecko로 확인 → 해당 시 🔴
  branch 2: 사이클 경과 < 6개월 → 해당 시 🟡
  branch 3: 사이클 정보 없음 → MCP 조회 실패 시 🟡
```

**data_sources 필드**: 규칙에 `data_sources`가 있으면, 해당 MCP 소스를 사용하여 조건 대입에 필요한 값을 조회하라. 이 필드는 **힌트**이며, 반드시 이 소스만 써야 하는 것은 아니다.

## 실행 B: KC(Kill Condition) 전제 추출 — 3단 역추적법

인과주장/예측에서 **숨겨진 전제(KC)**를 추출한다. 이것이 Logic Ground의 핵심이다.

### ★ KC 추출 깊이: 반드시 3단까지

LLM이 가장 빠지기 쉬운 함정은 **1단 전제만 추출하는 것**이다. 3단까지 파라.

```
[3단 역추적 예시]

문서 주장: "HBM 수요 증가로 SK하이닉스 영업이익 2배 성장 전망"

■ 1단 전제 (표면):
  KC-1: "HBM 수요가 실제로 증가하고 있는가?"
  → MCP 조회: HBM 출하량 YoY, AI 서버 CAPEX 추이

■ 2단 전제 (1단이 참이라고 해도):
  KC-2: "HBM 수요 증가 → SK하이닉스 수혜가 맞는가?"
  → 경쟁사 점유율 변화(삼성, Micron), ASP 추이, 수율 문제
  KC-3: "수요 증가가 '영업이익 2배'라는 규모로 전환되는가?"
  → 원가 구조, 감가상각, 환율 영향

■ 3단 전제 (2단이 참이라고 해도):
  KC-4: "이 성장이 지속 가능한가, 일회성이 아닌가?"
  → 고객사 재고 사이클, CAPEX 삭감 시그널, 대체 기술
```

### KC 추출 절차

1. **문서의 인과 체인을 분리**: A → B → C → 결론
2. **각 화살표(→)마다 질문**: "A가 참이라도, A→B가 성립하려면 무엇이 참이어야 하는가?"
3. **3단까지 반복**: 1단(표면) → 2단(전환 메커니즘) → 3단(지속 가능성/외부 조건)
4. **각 KC의 현재 상태를 MCP로 확인**
   ★ ① Fact에서 이미 조회한 MCP 값은 `{fact_evidence}`에 포함되어 있다.
   동일 시리즈(FRED, DART 등)를 재호출하지 마라. {fact_evidence}의 값을 먼저 확인하고,
   거기에 없는 데이터만 추가 MCP 호출하라.
5. **MCP 조회 불가하면 🟡 (미확인)으로 처리. 추정으로 참/거짓을 판단하지 마라.**

### ★ depends_on → KC 확장

claim에 `depends_on`이 있으면, **의존 claim의 KC도 이 claim의 전제 체인에 포함**하라.

```
예:
  c001 "HBM 수요 증가" → KC-1(수요 실제 증가?)
  c002 "HBM 수요로 이익 2배" depends_on: ["c001"]

  → c002의 KC 추출 시, c001의 KC-1도 c002의 전제 체인에 포함:
    c002 KC: KC-1(수요 증가, c001에서 상속) + KC-2(수혜 전환) + KC-3(규모 전환) + KC-4(지속성)
```

→ depends_on 체인이 깊으면, 최종 claim의 KC가 상위 claim의 KC를 포함하게 된다. 이것이 정상이다.

### KC 판정

| KC 상태 | 해석 |
|---|---|
| 참 (MCP 확인) | 전제 유효. 유효 조건으로 부착 |
| 거짓 (MCP 확인) | **KC 트리거** — 주장의 전제 붕괴. 🔴 |
| 미확인 (MCP 부재) | 🟡 부착. "이 전제가 바뀌면 결론이 무너진다"고 표기 |
| 약화 중 (방향은 맞지만 약해지는 중) | 🟡 부착. 추이와 임계점 표기 |

### ★ KC → 하류 전달

KC 결과는 후속 층에서 활용된다:

| 전달 대상 | 용도 |
|---|---|
| **Phase 3+4 finalize** | KC → validity_condition (유효 조건)으로 종합. kc_extracted 필드가 result_json에 포함되어 자동 전달 |
| **⑥ Omission** | KC 중 '약화 중' 또는 '미확인' 전제 → BBJ Break의 유력한 소재. kc_extracted에 기록하면 Omission 서브에이전트가 참조 |
| **④ Temporal** | KC 전제의 시점 유효성을 Temporal에서 재확인. 코드가 depends_on을 통해 자동 전파 |

## 실행 C: 연속 조건 주장 검증 (F-13 방어)

> F-13 방어규칙 (2026-03-17 추가)
> "X가 N일 연속 Y 이상" 같은 연속 조건 주장은 **최신값이나 평균이 아니라 각 날짜를 개별 검증**해야 한다.
> 하루라도 조건 미충족이면 연속이 끊기므로 결론이 달라진다.
>
> ★ 역할 경계: F-13은 **연속 조건의 논리적 성립 여부**를 검증한다.
> 변동률 수치의 정확성은 ① Fact의 F-11에서 이미 검증되었다.

### 연속 조건 주장 식별

아래 패턴의 주장이 발견되면 연속 조건 검증을 실행하라:

| 패턴 | 예시 |
|---|---|
| "N일/거래일 연속 X 이상/이하" | "VIX >25 5거래일 연속", "HYG 3일 연속 하락" |
| "N주/월 연속 상승/하락" | "3주 연속 자금 유출" |
| "X 이후 Y 지속" | "3/6 이후 25~28 구간 지속" |

### 검증 절차

```
1. 구간의 시작일과 종료일을 확정
2. MCP로 해당 구간의 일별(또는 주별) 데이터를 조회
3. 각 날짜마다 조건 충족 여부를 개별 확인
4. 조건 미충족 날짜가 하나라도 있으면:
   → "연속 [N]일" 주장은 사실 불일치
   → 미충족 날짜를 명시하고, 실제 연속 일수를 계산
   → 해당 claim을 🔴로 판정

예:
  주장: "VIX >25 3/6 이후 지속"
  MCP 조회:
    3/6: 29.49 ✅ | 3/9: 25.50 ✅ | 3/10: 24.93 ❌ | 3/11: 24.23 ❌ | 3/12: 27.29 ✅
  결과: 3/10~11에 25 미만. "3/6 이후 지속"은 사실 불일치.
    실제 연속: 3/12~3/13 = 2거래일.
  → 🔴 "F-13: 연속 조건 불일치. 3/10(24.93), 3/11(24.23)에 미충족."
```

### 연속 조건과 트리거 판정의 연결

연속 조건이 **다른 트리거의 전제**인 경우 (예: "VIX 5일 연속 >25" → L7 만성 트리거):
- 연속 조건이 사실 불일치 → 해당 트리거 판정도 과대 평가
- KC에 등록: `"premise": "VIX >25 5거래일 연속", "current_status": "실제 연속 2일. 카운트 리셋됨", "verdict": "🔴"`

### 탐색 대상 오류 패턴

| 패턴 | 설명 | 탐색 방법 |

| **후견 편향** | 과거를 보고 "당연한 결과"로 서술 | "예상대로 ○○했다" — 당시 예측이 실제로 있었는가? |

### ★ Fact F-12 편향 연계 확인

① Fact의 F-12에서 **기준점 선택 편향**이 플래그되었으면, 해당 claim에서 아래를 추가 확인하라:
- 기준점 편향이 **앵커링**(특정 유리한 수치에 고착)과 동반되는가?
- 기준점 편향이 **확증 편향**(유리한 방향으로만 기준점 선택)의 일부인가?
- F-12 수치 괴리를 Fact에서 플래그했으므로, Logic에서는 **"의도적 편향인지 단순 반올림인지"** 판단하라.

### 탐지 시 주의

- 패턴이 **의심**되면 notes에 기록하되, **단정하지 마라.** "확증 편향 가능성"이지 "확증 편향이다"가 아니다.
- 패턴 탐지 결과는 규칙 위반과 **별도로** 기록한다. rules_triggered에는 넣지 않고 notes에 서술.
- 2개 이상 패턴이 중첩되면 🟡 → 🔴 격상 고려.

## 실행 F: ACH 경쟁 가설 분석 (Analysis of Competing Hypotheses)

> KC는 "문서의 결론이 맞으려면 무엇이 참이어야 하는가"를 추출한다.
> ACH는 그 반대편: **"같은 증거로 다른 결론도 가능하지 않은가"**를 검증한다.

### ACH가 필요한 경우

아래 조건 중 **하나라도 해당하면** ACH를 실행하라:

- 인과주장 claim에서 **KC가 2개 이상 🟡(미확인)**일 때
- 문서의 결론이 **강한 방향성**(매수/매도/Bullish/Bearish)을 가질 때
- **확증 편향**이 실행 D에서 의심될 때

### ACH 실행 절차

```
Step 1. 가설 도출 (최소 3개)

  H1: 문서의 결론 (예: "HBM 수요로 SK하이닉스 영업이익 2배 성장")
  H2: 반대 가설 (예: "경쟁 심화로 마진 압축, 이익 정체")
  H3: 제3 가설 (예: "수요는 맞지만 공급 과잉으로 ASP 하락, 매출↑이익↓")

  ★ H2, H3를 만들 때 KC에서 추출한 전제를 활용하라.
    KC-2(점유율)가 거짓이면 → H2
    KC-3(수요→이익 전환)이 약하면 → H3

Step 2. 증거-가설 매트릭스 작성

  문서에 제시된 증거 + MCP로 추가 조회한 증거를 행(row)에 나열하고,
  각 증거가 각 가설을 지지(+) / 반박(-) / 무관(0)하는지 표기:

  | 증거 | H1 (문서 결론) | H2 (반대) | H3 (제3) |
  |---|---|---|---|
  | HBM 출하량 YoY +45% | + | 0 | + |
  | 삼성 HBM3E 수율 개선 뉴스 | 0 | + | + |
  | SK하이닉스 감가상각 YoY +30% | - | + | + |
  | AI CAPEX 가이던스 상향 | + | - | + |

Step 3. 진단 증거(Diagnostic Evidence) 식별

  진단 증거 = 가설 간 판별력이 높은 증거
  (한 가설은 +, 다른 가설은 -인 증거)

  위 예시에서: "삼성 HBM3E 수율 개선"은 H1(0) vs H2(+)를 구별하는 진단 증거.

Step 4. 판정

  - H1만 지지되고 H2/H3가 반박되면 → 문서 결론 유효 (Logic 판정에 반영 없음)
  - H2 또는 H3도 지지 가능하면 → notes에 "경쟁 가설 미배제" 기록
    → 🟡 격상 사유
  - H2/H3가 H1보다 더 많은 증거로 지지되면 → 🔴 격상 사유
```

### ACH 결과 기록

ACH 결과는 **notes에 텍스트로 기록**한다 (ach_result 필드는 코드에서 미지원).

```
notes에 포함:
  "ACH 실행: H1(문서 결론) support=3 contra=1 / H2(경쟁 심화) support=2 contra=1 /
   H3(매출↑이익↓) support=3 contra=0. 진단 증거: 감가상각 증가율.
   → H3 미배제. 🟡 격상 사유."
```

### ACH 실행하지 않는 경우

- 단순 사실 주장 (인과 체인 없음)
- KC가 모두 🟢(참)으로 확인된 경우
- 문서가 명시적으로 복수 시나리오를 제시한 경우 (이미 ACH와 유사)
