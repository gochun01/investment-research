# ① Fact Ground 검증 프롬프트

아래 claim의 수치/데이터를 **MCP 도구로 1차 소스를 직접 조회**하여 검증하라.

## 원칙

> 4대 불변 원칙, 판정 체계, 소스 신뢰 계층의 전체 정의는 **CLAUDE.md** 참조.
> 아래는 ① Fact Ground 전용 규칙만 기술한다.

## ① Fact 전용 원칙

- **MCP 미조회 🟢 금지** (V-01). LLM 지식으로 "맞는 것 같다"는 🟢이 아니다.
- **⚫→🟢 승격 금지**. 1차 소스가 없으면 ⚫ NO BASIS.
- 반드시 MCP 도구를 실제로 호출하여 값을 가져와야 한다.

## ★ 증거 신뢰도 계층 (ISA 500 기반)

모든 MCP 소스를 동등하게 취급하지 마라. 아래 **3단계 신뢰도 계층**에 따라 증거의 가중치가 다르다.

### 신뢰도 등급

| 등급 | 소스 유형 | 예시 | 판정 가중치 |
|---|---|---|---|
| **Tier 1 (직접 1차)** | 감사받은 재무제표, 중앙은행 공식 발표, 거래소 공시, 규제기관 공식 데이터 | DART 감사보고서, FRED(연준 직접), SEC EDGAR 10-K, 한은 기준금리 | 단독으로 🟢 판정 가능 |
| **Tier 2 (간접 1차)** | 데이터 어그리게이터, 뉴스 통신사, 미감사 속보, 실시간 시세 | Yahoo Finance, CoinGecko, CoinMetrics, DeFiLlama, Bloomberg 기사 | 단독 🟢 가능하나, 핵심 수치는 Tier 1 교차검증 권장 |
| **Tier 2 ★크립토 예외** | 온체인 지표(TVL, MVRV, 트랜잭션 등)는 감사된 Tier 1 소스가 원천적으로 존재하지 않음 | CoinMetrics, DeFiLlama, Etherscan | **크립토 온체인 한정 Tier 2 단독 🟢 허용** |
| **Tier 3 (2차/3차)** | LLM 사전학습 지식, 블로그, 커뮤니티, 애널리스트 추정치, 비공식 소스 | WebSearch 결과, SNS, 개인 블로그, 출처 불명 데이터 | **단독 🟢 불가**. Tier 1~2 교차검증 필수. 미검증 시 최대 🟡 |

### 적용 규칙

```
1. evidence 기록 시 각 소스의 tier를 표기하라 (권장):
   {"source": "DART", "tier": 1, "query": "...", "value": "...", "retrieved_at": "..."}
   ★ tier 필드는 코드에서 강제하지 않으나, self_audit(Phase 4.5)에서 Tier 3 의존 여부를 점검할 때 활용된다.
   notes에라도 "Tier 1 소스" / "Tier 2 소스" / "Tier 3 소스"를 명시하라.

2. Tier 3 소스만으로 검증된 claim → 최대 🟡 (PLAUSIBLE). 🟢 불가.

3. Tier 1과 Tier 2가 충돌할 때:
   → 원칙: Tier 1 우선. notes에 "Tier 2(Yahoo Finance)는 X이나, Tier 1(DART)은 Y. Tier 1 채택." 기록.
   → 예외: Tier 1이 노후(>30일)이고 Tier 2가 최신(≤7일)이면 → Tier 2 채택.
     notes에 "Tier 1 노후(N일), 최신 Tier 2 채택. Tier 1 업데이트 후 재검증 권장" 기록.

4. 동일 Tier 소스 간 충돌:
   → 차이 ≤5%: 🟡 처리. notes에 양쪽 값을 모두 기록하고 사용자 판단에 맡김.
   → 차이 >5%: 🔴 처리. notes에 "동일 Tier 소스 간 >5% 괴리. 데이터 신뢰성 의심." 기록.

5. 핵심 결론을 지탱하는 수치(목표가, 밸류에이션 핵심 입력):
   → 반드시 Tier 1 소스로 검증. Tier 2만 있으면 notes에 "Tier 1 미확인" 경고 부착.
```

### 증거 노후도와 Tier의 교차

| | Tier 1 | Tier 2 | Tier 3 |
|---|---|---|---|
| **최신 (≤7일)** | 🟢 가능 | 🟢 가능 | 🟡 최대 |
| **보통 (8~30일)** | 🟢 가능 (notes 부착) | 🟡 | 🟡 |
| **노후 (>30일)** | 🟡 | 🔴 | ⚫ |

### FRED 결측값 처리

일부 FRED 시리즈는 특정 기간에 결측값(".", "N/A", null)을 반환할 수 있다.

- **FEDTARMD**: FOMC SEP 발표 시점에만 값이 존재. 비발표 월은 "." 반환.
  → 가장 최근 유효값을 사용하고, 데이터 기준일을 반드시 기록하라.
- **분기/반기 시리즈** (GDP 등): 발표 지연으로 최신 분기 값이 아직 없을 수 있음.
  → "최신 가용 데이터"와 "문서 인용 기준일"의 차이를 Temporal에서 반영하라.

결측값 발생 시:
1. 다음 시리즈로 교차 확인 시도 (예: FEDTARMD 결측 → DFEDTARU 상한 활용)
2. 교차 확인도 불가하면 ⚫ NO BASIS 판정 (🟡로 넘기지 마라)

### 기타 MCP 소스 장애/결측 처리

| 소스 | 알려진 제한 | 대안 |
|---|---|---|
| **CoinGecko** | 무료 API 365일 제한. 365일 초과 시계열 요청 시 에러 | CoinMetrics 시계열로 대체 |
| **DeFiLlama** | 간헐적 타임아웃. 일부 프로토콜 데이터 지연 | Tavily로 DeFiLlama 페이지 스크래핑 |
| **DART** | 분기보고서 미제출 기업 존재. 사업보고서만 가용한 경우 있음 | Yahoo Finance 교차검증 |
| **SEC-EDGAR** | 한국 기업 미등록 | DART 단독 사용 |

공통 규칙: MCP 호출 실패 → **3회 재시도** → 대안 소스 시도 → 대안도 불가하면 **⚫ NO BASIS** (🟡로 넘기지 마라)

## 판정 기준

| 판정 | 조건 |
|---|---|
| 🟢 VERIFIED | 1차 소스와 ±2% 이내 일치 (금융 업계 반올림 관용치. 도메인별 조정 필요 시 rules.json에 추가) |
| 🟡 PLAUSIBLE | 2~5% 차이 또는 간접 증거만 존재 |
| 🔴 FLAGGED | 불일치 >5% 또는 방향 반대 |
| ⚫ NO BASIS | 검증할 1차 소스 자체 부재 |

### ★ evidence_type에 의한 판정 상한 제한

```
evidence_type = "estimate" (추정·전망 수치)인 claim:
  → 1차 소스와 ±2% 이내 일치하더라도 최대 🟡 (🟢 불가)
  → 이유: 추정치는 미래 불확실성이 내재. 현재 데이터와 일치해도 "검증됨"이 아님.
  → notes에 "estimate 하향: 추정치이므로 🟢→🟡" 기록

evidence_type = "fact" (확인된 데이터)인 claim:
  → 기본 판정 기준 그대로 적용

evidence_type = "opinion" (주관적 판단)인 claim:
  → Fact 층 대상이 아님 (claim_type_matrix에서 N/A). 이 층에 도달하지 않음.
```

## ★ 단위·기간·기준 정합성 체크 (비교 전 필수)

MCP 값과 비교하기 **전에** 아래를 반드시 확인하라. 단위가 다르면 비교 자체가 무의미하다.

| 항목 | 확인 | 불일치 시 처리 |
|---|---|---|
| **통화 단위** | 원 vs 달러 vs 유로 | 환율 변환 후 비교. 변환 기준일 명시 |
| **회계 기간** | 연간 vs 분기 vs TTM | 같은 기간끼리 비교. 기간 명시 |
| **연결 vs 별도** | 연결재무제표 vs 별도재무제표 | 반드시 같은 기준으로 비교 |
| **GAAP vs Non-GAAP** | 영업이익 vs 조정영업이익 | 기준 명시. 불일치 시 notes에 기록 |
| **K-IFRS vs US-GAAP** | 한국 기업과 해외 기업 수치 비교 시 회계기준 차이 주의 | 영업이익 정의(K-IFRS는 기타손익 포함 가능), 연결 범위, R&D 자본화 차이. notes에 "회계기준 차이 고려" 기록. lr_022 규칙 참조 |
| **산출 기준** | 보통주 기준 EPS vs 희석 EPS | 같은 기준으로 비교 |
| **시점** | claim의 시점 vs MCP 조회 시점 | 명백한 시점 차이는 notes에 표기 |

**정합성 불일치가 발견되면**: 비교 자체를 🟡로 처리하고, notes에 "단위/기간 불일치: [상세]"를 기록.

## ★ 시계열 주장 검증 (Time-Series Claim Verification)

> F-10, F-11, F-12 방어규칙 (2026-03-17 추가)
> "N일간 X% 변동"이나 "N일 연속 Y 이상" 같은 시계열 주장은 **현재값 대조만으로 불충분**.
> 기준점·구간·비거래일을 모두 검증해야 한다.

### 시계열 주장 식별

아래 패턴의 claim이 발견되면 시계열 검증을 실행하라:

| 패턴 | 예시 |
|---|---|
| "N일/주/월 변동 X%" | "Brent 30일 +50%", "DXY 1개월 +3.3%" |
| "N일 연속 X 이상/이하" | "VIX >25 5거래일 연속" |
| "A에서 B로 변동 (기간)" | "2월 중순 $68 → $105" |
| "N개월/분기 최고/최저" | "10개월 최고", "52주 범위" |

### 검증 1: 비거래일 방어 (F-10)

```
문서가 명시한 날짜가 비거래일(토·일·공휴일)이 아닌지 확인하라.

방법:
1. 문서의 데이터 시점 날짜를 추출
2. 해당 날짜가 토요일(6) 또는 일요일(0)이면 → 직전 금요일 종가를 의미
3. 미국 공휴일(Presidents Day, MLK Day 등)이면 → 직전 거래일

비거래일인데 거래일처럼 표기된 경우:
→ notes에 "⚠ F-10: [날짜]는 비거래일. 실제 데이터는 직전 거래일 [날짜] 종가" 기록
→ 값 자체가 맞으면 🟡 (날짜 오류만), 값도 다르면 🔴
```

### 검증 2: 시계열 구간 정합성 (F-11)

> ★ 역할 경계: F-11은 **변동률 수치의 정확성**만 검증한다.
> "N일 연속 X 이상"과 같은 **연속 조건의 논리적 성립 여부**는 ③ Logic의 F-13에서 검증한다.

```
"N일간 X% 변동" 주장에서 N일이 실제 시작·끝 날짜와 일치하는지 확인하라.

방법:
1. 시작점과 끝점의 구체적 날짜를 역산
   예: "30일 변동" → 끝점 3/13이면 시작점은 2/11~2/13
2. MCP로 시작점 날짜의 실제 값을 조회
3. 변동률을 직접 계산하여 문서의 변동률과 대조

오류 패턴:
- 시작·끝 사이가 N일이 아닌 경우 (13일 변동을 "30일"이라고 표기)
  → 🔴 "F-11: 실제 구간 [X]일, 문서 표기 [N]일. 구간 불일치."
- 변동률이 ±5%p 이상 차이
  → 🔴 "F-11: 실제 변동 [Y]%, 문서 표기 [X]%. 변동률 불일치."
- 구간은 맞으나 소수점 차이
  → 🟡
```

### 검증 3: 기준점 선택 편향 (F-12)

```
변동률/비교의 기준점(baseline)도 MCP로 검증하라. 현재값만 검증하면 불충분하다.

방법:
1. 문서가 비교 기준으로 사용한 과거 수치를 식별
   예: "2월 중순 ~$68" → 기준점
2. MCP로 해당 시점의 실제 값을 조회
3. 실제 기준점과 문서 기준점의 차이가 ±2% 초과 → 플래그

편향 탐지:
- 기준점이 실제보다 낮으면: 상승률 과대 (낙관 편향)
- 기준점이 실제보다 높으면: 하락률 과대 (비관 편향)
→ notes에 "⚠ F-12: 기준점 문서=[X], 실제=[Y]. 변동률 [문서Z% vs 실제W%]" 기록
→ 기준점 괴리 >5% → 🔴, 2~5% → 🟡

★ F-12는 "편향 탐지"이므로 ③ Logic에서도 재확인한다.
  Fact에서는 수치 괴리만 플래그, Logic에서 "의도적 편향인지" 판단.
```

## ★ 교차검증 (Cross-Validation)

하나의 수치에 대해 **2개 이상의 소스에서 조회 가능하면 교차검증**하라.

### 교차검증이 필요한 경우
- 핵심 결론을 지탱하는 수치 (목표가의 근거, 밸류에이션의 핵심 입력)
- 1차 소스의 값이 문서와 미묘하게 다를 때 (2~5% 회색지대)
- 1차 소스가 업데이트 지연이 의심될 때

### 교차검증 방법
```
예: "삼성전자 2025년 매출 300조"
  소스1: DART → 연결재무제표 → 매출 298조 (차이 0.7%)
  소스2: Yahoo Finance → Revenue → $223B → 환산 297조 (차이 1.0%)
  → 두 소스 모두 ±2% 이내 → 🟢 (교차확인됨)
```

### evidence에 교차검증 기록
```json
[
  {"source": "DART", "query": "재무제표", "value": "298조", "retrieved_at": "..."},
  {"source": "Yahoo Finance", "query": "Revenue", "value": "$223B (≈297조)", "retrieved_at": "..."}
]
```

## ★ 회색지대(2~5%) 처리 규칙

차이가 2~5% 사이일 때가 가장 판단이 어렵다. 아래 기준을 따르라:

| 상황 | 판정 | 이유 |
|---|---|---|
| 2~5% 차이 + 단위/기간 차이 가능성 | 🟡 | 비교 기준 자체의 불확실성 |
| 2~5% 차이 + 교차검증 시 두 소스 모두 같은 방향 | 🟡 (🔴 접근) | 문서의 수치가 과장/축소 가능성 |
| 2~5% 차이 + 반올림/추정 표기("약", "~") 있음 | 🟡 | 의도적 반올림으로 볼 수 있음 |
| 2~5% 차이 + 핵심 결론에 영향 | 🟡 (notes에 영향도 명시) | 5% 미만이라도 결론을 바꿀 수 있음 |

**핵심**: 회색지대에서는 판정을 단정하지 말고, **notes에 판단 근거를 상세히 기록**하여 사용자가 결정하게 하라.

## 실행 절차

1. claim의 수치를 식별 (예: "매출 10조원")
2. **단위·기간·기준 정합성 체크** (위 표 참조)
3. 적합한 MCP 도구 선택 (예: DART → 재무제표)
4. MCP 도구 호출하여 실제 값 조회
5. 리포트 값과 실제 값 비교 (정합성 확인 후)
6. **핵심 수치이면 교차검증** (2차 소스 조회)
7. **depends_on 체인 확인** (아래 규칙 참조)
8. 판정 + 근거 기록

### ★ depends_on 처리 규칙

claim에 `depends_on`이 있으면, 의존 claim의 Fact 판정을 확인하라.

```
예:
  c001 "매출 10조원" → Fact 🔴 (실제 8조)
  c002 "PER = 시총/매출 = 12배" depends_on: ["c001"]

  → c002의 PER 계산 근거(매출)가 틀렸으므로:
    c002 Fact 판정은 독립적으로 수행하되,
    notes에 "⚠ depends_on 경고: 의존 claim c001(매출)이 🔴.
    이 claim의 수치는 c001의 오류를 전제로 한 값." 부착.
```

판정 자체는 MCP 대조 결과 그대로 내리되(독립 검증 원칙), **의존 claim 오류의 파급 경고를 notes에 반드시 남겨라**.
→ 후속 층(③ Logic, ④ Temporal)에서 코드가 자동으로 🟡 전파한다.

## ★ Fact 조회 결과의 하류 전달

이 단계에서 MCP로 조회한 값은 **후속 층에서 자동 재사용**된다. 중복 호출하지 마라.

| 전달 대상 | 메커니즘 | 용도 |
|---|---|---|
| **④ Temporal** | `{fact_evidence}` 플레이스홀더로 코드가 자동 주입 | 기준 시점 비교. 동일 MCP 재호출 금지 |
| **③ Logic** | `{fact_evidence}` 플레이스홀더로 코드가 자동 주입 | KC 전제 확인 시 이미 조회된 값 재사용. 동일 시리즈 재호출 금지 |

★ LLM이 "기억"할 필요 없음 — **코드가 evidence를 구조화하여 후속 프롬프트에 자동 주입**한다.
