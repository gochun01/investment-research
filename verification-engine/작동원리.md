# verification-engine 작동 원리

> 문서의 수치·논리·누락을 MCP 1차 소스와 6층으로 대조한다.
> LLM은 판단하지 않는다. 대조 결과만 보고한다. 판단은 인간이 한다.

---

## 존재 이유

```
이 엔진 없이 반복되는 문제:
  LLM이 리서치/뉴스/계약서의 수치를 자체 판단으로 "맞다"고 단정한다.
  MCP 1차 소스와 대조하지 않고 내장 지식으로 🟢를 내린다.
  이것은 CFA V(A) "합리적 근거" 원칙을 위반한다.

이 엔진이 해결하는 핵심:
  "판단 없는 검증" — MCP 1차 소스와의 대조를 6층 구조로 강제한다.
```

**핵심 분업:**
- **코드는 "틀"** — 6층 순서, 스키마 검증, KC 상태 머신, 패턴 누적
- **지침은 "두뇌"** — LLM이 어떻게 판정할지의 기준과 절차
- **LLM은 "실행"** — 지침을 따라 실제 판정을 수행
- **JSON은 "기억"** — 세션이 끝나도 규칙·KC·패턴이 살아남음
- **MCP는 "손발"** — 1차 소스 데이터를 실시간 조회

## 설계 철학: "틀은 코드, 두뇌는 프롬프트"

```
코드가 정하는 것:                 LLM이 정하는 것:
━━━━━━━━━━━━━━━━                ━━━━━━━━━━━━━━━
6층 순서 (불변)                   각 층에서의 판정 (🟢🟡🔴⚫)
claim_type × layer 매트릭스       claim 추출 (어떤 주장이 있는가)
KC 상태 머신 (4단계)              KC 전제 내용 (무엇이 핵심 전제인가)
패턴 승격 기준 (3회+)             패턴 설명 (어떤 실패가 반복되는가)
HTML 보고서 골격                  보고서 코멘트와 해석
스키마 검증 규칙                  수집 전략과 도구 선택

→ 프롬프트만 바꾸면 판정 기준이 바뀌고
→ 코드를 건드리지 않아도 됨
```

## 6층이 관통하는 원리

```
이 엔진의 핵심 가치는 "한 두뇌가 6층을 관통하며 연결을 발견하는 것"이다.

예: 천궁-II 검증에서의 관통:

  Fact (c009): PSR 35배 → 🟡 (실제로는 P/E일 가능성)
      ↓
  Logic (c009): PSR 35배로 과열 판정 → 🟡 (지표 혼용이 판단 왜곡)
      ↓
  KC 추출: "PSR 35배가 P/E 오기이면 밸류에이션 해석 완전히 달라짐"
      ↓
  Omission (문서): 이 혼용 가능성이 보고서에서 언급 안 됨 → 🟡

  = Fact의 🟡이 Logic의 🟡을 유발하고
    Logic에서 KC가 추출되고
    Omission에서 그 KC가 누락으로 잡힘

이것을 3개 서브로 쪼개면?
  서브A(Fact): "35배 확인" → 🟡
  서브B(Logic): Fact 결과를 모름 → KC 추출 불가
  → 관통이 끊어짐. 그래서 6층은 절대 병렬 불가.
```

## 핵심 데이터 구조

### claim_type × layer 매트릭스

```
                Fact  Norm  Logic  Temporal  Incentive  Omission
수치주장         ✅    ❌    ✅      ✅         ❌         ❌
인과주장         ❌    ❌    ✅      ✅         ❌         ✅
예측             ❌    ❌    ✅      ✅         ❌         ✅
사실진술         ✅    ❌    ❌      ✅         ❌         ❌
의견             ❌    ❌    ❌      ❌         ✅         ❌
조항             ❌    ✅    ✅      ❌         ✅         ✅

→ 수치주장에는 Norm/Incentive/Omission을 적용하지 않는다 (과검증 방지)
→ 의견에는 Fact를 적용하지 않는다 (의견은 "맞다/틀리다"가 아님)
```

### KC (Kill Condition) 생명주기

```
active → approaching → resolved → revived
  │          │            │           │
  │      proximity      트리거      재활성화
  │      85%+           돌파        조건 변경
  │
  "금리 4.5% 이상 유지 시 DCF 할인율 전제 무효"
  → 현재 4.3% (proximity 86%) → approaching
  → 4.5% 돌파 시 → resolved (관련 검증 재검증 필요)
```

### 패턴 레지스트리

```
flag (1회) → candidate (2회) → proposed (3회+) → promoted / dismissed
  │              │                  │               │
  첫 발견        반복 확인           규칙 승격 제안    rules.json에 등록
                                                    또는 기각
```

```
  단기: state/current-status.json (세션 현황)

```

## 불변 원칙 (CLAUDE.md)

```
1. 판정의 전제: MCP 1차 소스와 대조한 결과만 보고하라.
   판단 금지. "이 보고서는 신뢰할 수 있습니다" ← 금지.

2. 판정 불가 시 원칙: 기준 데이터가 없으면 ⚫ NO BASIS.
   "아마 맞을 것 같다"로 🟢를 내지 않는다.

3. 판정 비확정 원칙: 하나의 해석을 정답으로 선언하지 않는다.
   🟡 = "이런 근거로 이런 상태이다"까지만.

4. 수정 제안 원칙: 확정 / 권장 / 경고만(Advisory) 3단계.
```

## 자율 판단 보고서 (render_adaptive.py) 작동 원리

### 핵심 철학

```
기존 html_renderer.py의 문제:
├── 항상 7+1 섹션 고정 → 데이터 적은 층도 빈 섹션 생성
├── 모든 검증에 같은 골격 → 🟢 전체 통과와 🔴 위기가 같은 형태
└── 결과: 형식이 내용을 지배

render_adaptive.py의 원칙:
├── 검증 결과를 먼저 읽는다
├── 결과의 무게 중심을 찾는다 (어떤 층에 🔴/🟡가 집중?)
├── 무게 중심에 맞게 구조를 설계한다
├── 데이터 없는 층은 섹션을 만들지 않는다
└── 결과: 내용이 형식을 결정함
```
