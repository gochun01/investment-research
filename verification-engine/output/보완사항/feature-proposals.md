# Verification Engine — 기능 제안서

작성일: 2026-03-19

---

## 1순위: 즉시 효과

### A. 검증 이력 로컬 저장

현재 `verify_finalize()`는 JSON을 반환만 하고 끝. 세션 종료 시 결과 소실.

- `output/history/` 폴더에 `vrf_20260319_*.json` 자동 저장
- 이전 검증 결과 조회 도구: `verify_list_history()`, `verify_load_history(vrf_id)`
- 같은 문서 재검증 시 이전 결과와 자동 diff

**난이도:** 낮음 | **효과:** 높음

---

### B. E2E 테스트 스위트

검증 엔진 코드는 있지만 실전 테스트 0건. 실제 문서로 파이프라인을 돌려본 적이 없음.

- `tests/` 폴더에 샘플 문서 3~5개 (equity, crypto, macro, legal, geo)
- 각 문서에 대한 예상 claim 수, 예상 판정, 예상 Finding 수 정의
- `pytest` 또는 스크립트로 Phase 0→5 전체 흐름 자동 검증

**난이도:** 중간 | **효과:** 높음

---

## 2순위: 품질 향상

### C. 신뢰 점수 (Confidence Score)

현재 판정은 4등급뿐. 같은 YELLOW라도 "거의 GREEN"과 "거의 RED"이 구분 안 됨.

- 6층 각각에 0~100 점수 부여
- 가중 평균으로 종합 점수 산출 (예: 72/100)
- Fact/Logic 가중치 높게, Incentive 낮게 등 doc_type별 가중치 차등

**난이도:** 중간 | **효과:** 중간

---

### D. 문서 버전 비교 (Diff Verification)

같은 문서의 v1 → v2를 비교 검증. 리서치 보고서 업데이트, 계약서 수정 시 유용.

- v1 검증 결과 로드 → v2 claim 추출 → 변경된 claim만 재검증
- "수정됨 / 신규 / 삭제됨 / 유지" 태그
- HTML 보고서에 DELTA 섹션 추가

**난이도:** 높음 | **효과:** 높음

---

### E. KC 모니터링 (자동 재검증 트리거)

`finalize`에서 추출한 `invalidation_triggers`를 실제로 감시.

- 예정 이벤트 날짜 도래 시 알림 (FOMC, 실적발표 등)
- 해당 KC의 MCP 데이터를 자동 재조회하여 "유효/약화/트리거" 판정
- `verify_check_triggers(vrf_id)` 도구 추가

**난이도:** 중간 | **효과:** 높음

---

## 3순위: 확장성

### F. 멀티 문서 교차 검증

현재는 단일 문서만 검증. 같은 주제의 복수 리포트를 교차 비교하면 편향 탐지 강화.

- 동일 종목 리서치 3건 → 목표가/전제/KC 비교
- 합의(consensus) vs 이탈(outlier) 자동 식별
- HTML SECTION VIII(부록)에 교차 비교 매트릭스

**난이도:** 높음 | **효과:** 중간

---

### G. 사용자 피드백 루프

검증 결과에 대한 사후 추적 — "이 판정이 맞았나?"

- 3개월 후 실제 결과와 비교: "목표가 10만원 → 실제 8만원 → Fact GREEN but 결론 RED"
- 오답 패턴 축적 → 규칙/체크리스트 자동 개선 제안
- `verify_record_outcome(vrf_id, actual_result)` 도구

**난이도:** 높음 | **효과:** 높음

---

### H. 요약 보고서 유형 추가

현재 HTML 보고서 1종. 용도별 경량 포맷 추가.

- Telegram 카드: 판정 + KC + 액션 3줄 (텍스트)
- Slack Block: 구조화된 JSON Block Kit
- PDF 내보내기: HTML → PDF 변환 (인쇄 CSS 활용)

**난이도:** 낮음 | **효과:** 중간
