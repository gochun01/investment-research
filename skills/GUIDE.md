## 이 스킬이 하는 일

**한 줄 요약:** Claude가 어떤 투자 대상이든 기관 투자자 수준의 구조로 분석하도록 지시하는 프레임워크입니다.

세 개의 파일이 함께 작동합니다.

| 파일 | 비유 | 역할 |
|---|---|---|
| `SKILL.md` | 뇌 | 분석 흐름 결정, 질문 구조화, 집계 규칙 |
| `SECTIONS.md` | 근육 | 각 섹션을 어떻게 채울지 정의하는 스펙 |
| `EXAMPLES.md` | 거울 | 실제 output이 어떻게 생겼는지 보여주는 샘플 |

Claude는 분석 요청을 받으면 세 파일을 동시에 참조하여 리포트를 생성합니다. 파일을 직접 읽거나 편집할 필요는 없습니다. 호출만 하면 됩니다.

## 바로 시작하는 법

### 가장 간단한 호출 — "풀스택으로"

```
"[대상] 풀스택으로 분석해줘"
```

예: `"LINK 풀스택으로 분석해줘"`

→ Claude가 Step 0을 실행합니다. 이 대상에 특화된 서브 질문 후보 4개를 제안하고, 선택을 받은 후 전체 7섹션 리포트를 생성합니다. MCP 전체 연동, Notion 저장까지 포함됩니다.

---

### 서브 질문을 직접 입력하면 Step 0을 건너뜁니다

```
"삼성전자 풀스택으로 분석해줘.
Q1-a: HBM 수익이 메모리 사이클 하강에도 방어 가능한가?
Q2-a: TSMC 대비 파운드리 고객 전환 비용이 실질적으로 존재하는가?"
```

→ 제안 단계 없이 즉시 분석 시작. 대상을 이미 잘 아는 경우 유용합니다.

---

### 빠르게 훑을 때

```
"ETH 빠르게 분석해줘"
```

→ MCP 없음. 웹서치만으로 핵심 3섹션(시장·수익·밸류에이션) 즉시 생성. Notion 저장 없음. 방향 탐색용이며 🟢 데이터가 없어 의사결정 근거로는 부족합니다.

---

### 특정 섹션만 필요할 때

```
"LINK 수익 구조만 분석해줘"      → Section 3만
"NVDA 밸류에이션 모델 만들어줘"  → Section 5만
"이 분석 검증해줘"               → 기존 리포트에 신뢰도 태그 부착
```

## 리포트 구조 한눈에 보기

```
Section 0 │ Snapshot & Variant View      ← 마지막에 작성, 맨 앞에 출력
          │ 핵심 수치 7개 + "왜 지금인가"

Section 1 │ Market & Competitive Landscape   ← Q2(해자) 판단의 병목
          │ TAM/SAM + 경쟁 매트릭스

Section 2 │ Business / Protocol Model
          │ 돈의 흐름 + 고객 전환 비용

Section 3 │ Financials & KPIs / Onchain Metrics   ← Q1(수익) 판단의 병목
          │ 수익 3조건 (실재 · 성장 · 귀속)

Section 4 │ Tokenomics 2.0  (크립토)
          │ Capital Allocation  (기업)

Section 5 │ Valuation & Scenario
          │ 멀티플 + DCF + 민감도 분석

Section 6 │ Key Risks & Mitigants
          │ 리스크 매트릭스 + 시나리오 전환 임계값

Section 7 │ Recommendation & Catalysts
          │ 투자 의견 + 촉매 타임라인 + Q1·Q2 최종 답변
```

**작성 순서 ≠ 출력 순서.** Claude는 Section 1→7 순서로 분석하고, Section 0(Snapshot)을 마지막에 작성하여 맨 앞에 붙입니다. 결론을 먼저 쓰면 확증편향이 생기기 때문입니다.

## 설계 의도와 기대 효과

### 의도 1 — 보편 질문 + 서브 질문의 혼합 구조

이 프레임워크는 두 개의 보편 질문으로 수렴합니다.

> **Q1.** 이 대상은 없어지지 않고, 지속 가능한 방식으로 돈을 버는가?
> **Q2.** 이 대상은 경쟁자 대비 대체 불가능한 해자가 무엇인가?

LINK든 삼성전자든 ETH든 동일한 질문을 적용합니다. 덕분에 분석 대상이 달라져도 리포트의 뼈대가 흔들리지 않습니다.

서브 질문은 이 보편 질문을 대상에 맞게 날카롭게 만드는 역할입니다.

```
Q1 + Q1-a: "L1 수수료 수익이 L2 성장에 잠식되고 있는가?"
→ Section 3의 분석이 L1 vs L2 fee 비교로 집중됨
```

서브 질문이 없어도 분석은 가능합니다. 대상을 잘 모를수록 Claude가 제안하는 후보 중 하나를 선택하는 것이 낫습니다.

**기대 효과:** 어떤 대상을 분석해도 같은 판단 기준이 적용되므로 자산 간 비교가 가능해집니다.

---

### 의도 2 — 섹션이 연결된 검증 트리

섹션들은 단순 나열이 아니라 인과로 연결되어 있습니다.

```
Section 3 수익 추정치  →  Section 5 DCF 입력값
Section 5 Bear 목표가  →  Section 6 트리거 기준선
Section 6 리스크 임계값  →  Section 7 청산 조건
```

Section 3에서 수익 논거가 미성립(❌ 2개 이상)이면 Section 4(토크노믹스)는 생략되고, Section 5는 Bear 시나리오만 생성되며, 투자 의견은 Underweight로 확정됩니다. 분석 흐름 자체가 데이터에 의해 결정됩니다.

**기대 효과:** "수익이 없는 프로토콜인데 Overweight" 같은 구조적 모순이 생기지 않습니다. 결론이 앞 섹션의 논거와 항상 연결됩니다.

---

### 의도 3 — Variant View, 컨센서스와 다른 점을 먼저 밝히는 원칙

Section 0(Snapshot)에는 수치 박스와 함께 Variant View가 있습니다.

```
컨센서스: 시장이 일반적으로 믿는 것 (1문장)
내 견해:  내가 다르게 보는 것 (1문장)
근거:     어느 섹션의 데이터가 이 차이를 지지하는가
리스크:   내가 틀릴 경우 (1문장)
```

Variant View가 없으면 리포트는 컨센서스를 정리한 것에 불과합니다. 초과 수익은 시장과 다르게 보는 지점에서 나옵니다.

**기대 효과:** 리포트를 읽는 사람이 "이 분석이 시장 대비 어디에 베팅하는가"를 30초 안에 파악할 수 있습니다. 분석의 핵심 논거가 흐릿해지지 않습니다.

---

### 의도 4 — 이중 신뢰도 태그, 데이터 품질을 결론과 분리

리포트의 모든 수치와 결론에는 두 종류의 태그가 붙습니다.

**Layer 1 — 소스 품질** (수치 옆에 표시)

| 태그 | 의미 |
|---|---|
| 🟢 | MCP 직접 호출·공식 공시 |
| 🟡 | 웹서치·추정·1차 보도 |
| 🔴 | 단일 출처·미검증 |
| ⚫ | 출처 불명·가정값 |

**Layer 2 — 결론 확신도** (섹션 결론 문장 끝에 표시)

| 태그 | 의미 |
|---|---|
| 🄰 | 높은 확신 — 🟢 소스 2개 이상 수렴 |
| 🄴 | 조건부 확신 — 명시된 가정이 있음 |
| 🅄 | 낮은 확신 — 데이터 불충분 |
| ● | 판단 보류 — 상충 데이터 존재 |

**🔴 데이터 + 🄰 결론은 금지 조합**입니다. 이 조합이 보이면 그 섹션의 논거를 먼저 의심하세요.

**기대 효과:** 리포트를 다 읽지 않아도 태그만 훑으면 어디가 약한 고리인지 즉시 파악됩니다. 또한 Claude가 확인되지 않은 수치를 확신 있는 결론으로 포장하는 것을 구조적으로 막습니다.

## Q1·Q2 집계 읽는 법

Section 7 끝에 다음 형태로 최종 판단이 나옵니다.

```
Q1: 조건부 Yes — 🄴 | 병목: Section 3 (🄴)
Q2: 전통 금융 파트너십 해자 — 🄰 | 병목: Section 1 (🄰)
```

세 가지를 확인하세요.

1. **병목 섹션** — 그 섹션의 태그가 Q1 또는 Q2의 상한입니다. Section 3이 🅄이면 Q1은 절대 🄰가 될 수 없습니다.
2. **"조건부"** — 뒤에 붙는 전제 조건이 깨지면 판단 전체가 바뀝니다. 이 조건을 모니터링하는 것이 포지션 관리의 핵심입니다.
3. **서브 질문 답변** — Q1-a, Q2-a가 Q1·Q2보다 더 구체적인 판단을 담고 있습니다. 이것을 먼저 읽는 것이 효율적입니다.

## 자주 하는 실수

**"빠르게"를 풀스택처럼 기대하는 것**
빠르게 모드는 MCP를 쓰지 않아 🟢 데이터가 없습니다. 중요한 결정 전에는 반드시 풀스택을 사용하세요.

**서브 질문 없이 Claude 제안도 생략하는 것**
Step 0에서 서브 질문을 생략하면 보편 구조로 진행됩니다. 분석이 넓어지지만 대상의 핵심 쟁점을 파고들지 못할 수 있습니다. 대상을 잘 모를수록 Claude 제안 중 하나를 선택하는 편이 낫습니다.

**Appendix를 읽지 않는 것**
Appendix B(검증 트리 실행 로그)에 "섹션별 결론 태그, 생략된 모듈, Q1·Q2 집계 과정"이 압축 정리되어 있습니다. 리포트 전체를 읽기 전에 Appendix B를 먼저 보면 어느 섹션에 집중할지 빠르게 판단할 수 있습니다.
