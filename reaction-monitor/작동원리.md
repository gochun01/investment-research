# reaction-monitor 작동 원리

> 쟁점이 들어오면 → 반응이 나타날 채널을 자율 선정 → 사실만 수집 → 계층 간 패턴을 읽고 → 미해소를 추적한다

```
┌──────────┬──────────┬────────────────────────────────────┐
│ 요소      │ 비유      │ reaction-monitor에서의 역할          │
├──────────┼──────────┼────────────────────────────────────┤
│ 코드      │ 뼈대      │ validate.py, render.py,            │
│ (Python)  │          │ watch.py, events.py                │
│          │          │ → 검증, 렌더링, 추적을 자동화        │
├──────────┼──────────┼────────────────────────────────────┤
│ 지침      │ 규칙      │ CLAUDE.md (R-01~R-11),             │
│ (MD)     │          │ SKILL.md (Phase 0~6),              │
│          │          │ GUARDRAILS.md (Green/Yellow/Red)    │
│          │          │ → LLM이 무엇을 어떻게 할지 결정      │
├──────────┼──────────┼────────────────────────────────────┤
│ LLM      │ 두뇌      │ Claude                              │
│          │          │ → 채널 선정, 수집, 판독, 코멘트       │
│          │          │ → 지침을 따르되 자율적으로 추론       │
├──────────┼──────────┼────────────────────────────────────┤
│ JSON     │ 기억      │ state.json, active-watches.json,    │
│          │          │ events/*.json, system-issues.json,   │
│          │          │ history/*.json                       │
│          │          │ → 세션 간 브릿지. 어제를 기억         │
├──────────┼──────────┼────────────────────────────────────┤
│ MCP      │ 손발      │ Yahoo Finance, FRED, CoinGecko,     │
│          │          │ WebSearch, Firecrawl, Tavily,        │
│          │          │ DeFiLlama, CoinMetrics, DART, SEC    │

└──────────┴──────────┴────────────────────────────────────┘

**핵심 분업:**
- **코드는 "일관성"을 보장** — 스키마 검증, 중복 방지, HTML 형식 고정
- **LLM은 "적응성"을 담당** — 쟁점마다 다른 채널 선정, 판독, 코멘트
- **JSON은 "연속성"을 담당** — 세션이 끝나도 상태/Watch/이벤트가 살아남음

## 자율 3조건

### 1. 기억 (Memory) — "어제를 안다"

```
세션 A: 나프타 수집 → state.json 저장 → 세션 종료 → LLM 소멸
                                          │
세션 B: 이란 원유 수집 → state.json 읽기 ←─┘
        "이전 미해소 UQ-002(수입선 다변화)가 이번 쟁점과 연결"
        → UQ-002 부분 해소 기록
                                          │
세션 C: 에틸렌 수집 → state.json 읽기 ←───┘
        "이전 나프타 UQ-003(하류 도미노)이 실제로 시작"
        → UQ-003 부분 해소 기록

기억이 없으면:
  매 세션이 처음부터 시작. 연쇄를 모름. 이전 미해소를 잊음.

기억이 있으면:
  세션이 바뀌어도 쟁점의 흐름이 이어짐.
```

기억의 계층:

```
단기 기억:  state.json          → "지금 어떤 쟁점이 열려있는가"
중기 기억:  active-watches.json → "무엇을 언제 체크해야 하는가"
장기 기억:  events/*.json       → "이 쟁점이 어떻게 전개되었는가"
메타 기억:  system-issues.json  → "시스템 자체의 약점은 무엇인가"
```

### 2. 루프 (Loop) — "자동으로 돌아간다"

```
감지 ──→ 판단 ──→ 실행 ──→ 검증 ──→ 기억 갱신 ──→ 다음 감지
  │        │        │        │         │              │
Phase 0  Phase 1~2 Phase 3  Phase 4  Phase 5~6      Phase 0
Watch    채널선정   수집     판독     Watch/Event    다음세션
스캔     지문추론   MCP호출  4렌즈    등록            자동스캔
```

루프의 핵심: **Phase 0이 매 세션 시작 시 자동 실행.** 사용자가 "Watch 체크해줘"라고 안 해도 기한 도래 Watch를 알려줌.

### 3. 울타리 (Guardrail) — "폭주하지 않는다"

```
┌─────────────────────────────────────────────────────────┐
│                    울타리 3단계                            │
│                                                           │
│  Green (무승인):                                          │
│    읽기, MCP 수집, 스캔, 계산                              │
│    → LLM이 자유롭게 실행. 부작용 없음.                     │
│                                                           │
│  Yellow (승인 필요):  ← 기본 정책                         │
│    state.json 갱신, Watch 등록/종료, 이벤트 생성,          │
│    HTML 생성, 이슈 적재, 설계 파일 수정                    │
│    → "실행할까요?" 확인 후 진행                            │
│                                                           │
│  Red (금지):                                              │
│    투자 판단, 파일 삭제, 매체/전문가 영구 판정              │
│    → 어떤 상황에서도 실행 불가                             │
└─────────────────────────────────────────────────────────┘
```

## "구조화된 자율성" 원리

```
고정 (코드 + 스키마가 강제):         자율 (LLM이 추론):
━━━━━━━━━━━━━━━━━━━━━            ━━━━━━━━━━━━━━━━━━━
반응 계층 5개                      계층 안의 구체적 채널
지문 차원 5개                      차원의 구체적 값
판독 렌즈 4개                      렌즈로 본 구체적 판독
state.json 필드                   필드 안의 값
Watch resolve_type 4종            Watch의 구체적 스케줄
이벤트 timeline 구조               timeline 안의 내용
validate.py 검증 규칙 11개         수집의 구체적 내용
HTML template 구조                template 안의 데이터
울타리 G/Y/R Zone                 Zone 안에서의 행동
```

**고정이 일관성을 보장하고, 자율이 적응성을 제공한다.** 같은 state.json 스키마를 쓰므로 시계열 비교가 가능하고, 다른 채널을 선정하므로 쟁점마다 적합한 수집이 된다.

## 코드 5개의 역할

```
validate.py               "state.json이 올바른가?"
  ├── 스키마 검증 11개 규칙 (R-01~R-09 반영)
  ├── WARN → system-issues.json 자동 적재 (--auto-log)
  └── 중복 방지

render.py                  "state.json을 고정 구조로 보여주기"
  ├── template.html 로드 → §1~§5 {{플레이스홀더}} 치환
  └── reports/*-reaction.html 생성

render_adaptive.py         "state.json을 데이터가 결정하는 구조로 보여주기" ★
  ├── Phase 1: 데이터 읽기 (5가지 추출)
  ├── Phase 2: 구조 설계 (유형·섹션 자율 결정)
  ├── Phase 3: 컴포넌트 조합 → HTML
  ├── Phase 4: V1~V5 자기 검증
  └── reports/*-adaptive.html 생성

watch.py                   "미해소를 잊지 않게"
  ├── propose: unresolved → Watch 제안
  ├── register: 승인 후 등록
  ├── scan: 기한 도래 스캔
  └── status: 현황 출력

events.py                  "쟁점의 생애를 추적"
  ├── create: 이벤트 생성 (+ parent 연결)
  ├── link: 후속 분석 연결
  ├── list/view: 조회
  └── chain: 연쇄 트리 출력
```

**LLM은 이 코드를 호출하지 않는다. 코드는 LLM의 결과물(state.json)을 후처리한다.** LLM이 수집과 판독을 하고, 코드가 검증·렌더링·추적을 한다. 관심사가 분리되어 있다.

## 자율 판단 보고서 (render_adaptive.py) 작동 원리

### 핵심 철학

```
기존 render.py의 문제:
├── 데이터가 적은데 §1~§5 고정 → 빈 섹션, 억지 채움
├── 데이터가 한쪽에 집중인데 균등 배분 → 핵심 희석
├── 이벤트 성격이 다른데 같은 틀 → 맞지 않는 옷
└── 결과: 형식은 완벽하나 내용이 형식을 따라감

render_adaptive.py의 원칙:
├── 데이터를 먼저 읽는다
├── 데이터의 무게 중심을 찾는다
├── 무게 중심에 맞게 구조를 설계한다
├── 데이터가 부족한 영역은 과장하지 않고 생략한다
└── 결과: 내용이 형식을 결정함
```

### 4-Phase 파이프라인

```
state.json
    │
    ▼
Phase 1: 데이터 읽기 ─────────────────────────────────
│  state.json에서 5가지만 추출한다:
│
│  ① Core Claim (핵심 주장)
│     pattern.direction_rationale에서 1문장.
│     → 보고서의 첫 줄이 된다.
│
│  ② Tension (긴장)
│     direction_detail에서 충돌하는 힘 감지.
│     ↑(호재)와 ↓(리스크)가 공존? 또는 ↑↓(분열)?
│     → 있으면 대립형, 없으면 서사형.
│
│  ③ Gravity (무게 중심)
│     각 반응 계층의 데이터 양을 측정.
│     (항목 수 × 10 + 내용 깊이)로 가중치 계산.
│     → 무게 순서가 섹션 순서를 결정.
│     예: narrative(120) > expert(95) > positioning(60) > ...
│
│  ④ Timeline (시간축)
│     fingerprint.time_character에서 과거/현재/미래 판단.
│     미해소 3건+ → 미래 비중 높임.
│
│  ⑤ Unresolved (미해소)
│     open 상태 미해소 건수.
│     → 있으면 보고서 말미에 포함.
│
│  출력 예:
│    Core Claim: "5개 계층이 호재 방향으로 수렴"
│    Tension: 있음 — positioning 내부 분열
│    Gravity: narrative > expert > positioning > pattern > policy > price
│    Timeline: 현재+미래
│    Unresolved: 4건

### render.py와 render_adaptive.py 비교

```

### 불변 규칙 (자율 보고서)

```

## 이벤트 체인이 보여주는 것

```
EVT-나프타 (수렴 — 모든 계층 위기)
  │
  ├── EVT-이란 원유 (분열 — 긍정 벡터 투입)
  │     "수렴(위기)이 분열로 전환. 30일 한시 한계."
  │
  └── EVT-에틸렌 하류 (분열 — 매체와 시장의 괴리)
        "매체는 '일파만파'지만 시장은 '조선만 진짜'."

EVT-크립토 (괴리 — SEC 호재↑ vs 가격↓)
  "전쟁이 호재의 가격 전파를 차단."
```

이벤트 체인은 **"패턴의 변화"를 추적한다.** 나프타가 "수렴"에서 시작해서 이란 원유가 "분열"로 바꾸고, 에틸렌이 "매체 vs 시장 괴리"를 보여준다. 하나의 쟁점이 시간에 따라 어떻게 전개되는지를 한 눈에 볼 수 있다.

## 전체 도구 체인

```
수집 완료 후:
  python core/pipeline.py --adaptive   → 전체 실행 (검증 → 감사 → 자율보고서 → Watch)
  또는 개별 실행:
    python core/validate.py --auto-log   → 스키마 검증 + WARN 이슈 적재
    python core/render_adaptive.py       → 자율 판단 HTML 보고서 생성 (★ 권장)
    python core/render.py                → 고정 구조 HTML 보고서 생성
    python core/watch.py propose         → 미해소 → Watch 제안
    python core/watch.py register        → (승인 후) Watch 등록
    python core/events.py create         → (승인 후) 이벤트 생성

다음 세션 시작 시:
  python core/watch.py scan            → 기한 도래 Watch 알림
  python core/validate.py --auto-log   → 상태 검증
  python core/events.py chain          → 이벤트 연쇄 확인
```
